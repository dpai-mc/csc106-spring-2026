Network Basics Check

CSC 106 – Introduction to Linux II  ·  Week 6 Lab  ·  15 points  ·  CSLO 2

Scenario

You are a junior systems administrator at a small tech company. Before handing a server off to the network team, your lead asks you to perform a quick network audit: confirm the machine has a valid IP address, can reach the internet, resolves DNS correctly, and is only exposing the services it should be. This kind of pre-deployment check is routine any time a server is freshly installed or reconfigured.

In this activity you will run five standard diagnostic commands on your Ubuntu Server VM, record the actual output, and explain in plain language what each result reveals about your system's network configuration.

By completing this activity you will demonstrate that you can:
  • Use ip addr to identify your VM's network interfaces and assigned IP address.
  • Use ip route to locate the default gateway.
  • Use ping to verify basic internet connectivity.
  • Use dig to confirm that DNS name resolution is working.
  • Use ss -tlnp to identify which network services are listening for connections.
Phase 1 of 5

Inspect Network Interfaces

Every networked device has at least one network interface — a logical connection between the OS and a physical or virtual network adapter. The ip addr command lists all interfaces and the IP address assigned to each. Your VM should have at least two: the loopback interface (lo), used only for internal OS communication, and a primary interface (often named enp0s3, ens3, or similar) that connects to your host machine's network via VirtualBox or UTM.

Run this command 
ip addr
Sample output — yours will differ 
1: lo: <LOOPBACK,UP,LOWER_UP> mtu 65536 qdisc noqueue state UNKNOWN
    link/loopback 00:00:00:00:00:00 brd 00:00:00:00:00:00
    inet 127.0.0.1/8 scope host lo         ← loopback, always 127.0.0.1
    inet6 ::1/128 scope host

2: enp0s3: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 state UP
    link/ether 08:00:27:4a:b1:c3 brd ff:ff:ff:ff:ff:ff
    inet 10.0.2.15/24 brd 10.0.2.255 scope global dynamic enp0s3  ← your VM's IPv4 address
    inet6 fe80::a00:27ff:fe4a:b1c3/64 scope link
What to look for
  • Find the interface that is not lo — that is your primary network interface.
  • On that interface, look for a line starting with inet (not inet6). The address shown (e.g., 10.0.2.15) is your VM's IPv4 address.
  • The /24 after the address is the subnet prefix length — it corresponds to a 255.255.255.0 subnet mask.
  • VirtualBox NAT mode typically assigns addresses in the 10.0.2.x range. UTM may use 192.168.x.x.
  • If you see no inet line on your primary interface, your VM may not have received a DHCP address. Check that the network adapter is enabled in VirtualBox / UTM settings.
Phase 2 of 5

View the Default Route

An IP address tells your VM who it is on the network, but not how to send traffic to machines outside its own subnet. For that it needs a default gateway — a router that forwards packets toward the internet. The ip route command displays the routing table: the set of rules that determine where packets go based on their destination address. The most important entry is the default route (default via …), which handles any destination that does not match a more specific rule.

Run this command 
ip route
Sample output — yours will differ 
default via 10.0.2.2 dev enp0s3 proto dhcp src 10.0.2.15 metric 100  ← default gateway
10.0.2.0/24 dev enp0s3 proto kernel scope link src 10.0.2.15  ← local subnet (reached directly)
What to look for
  • The line starting with default via is the key entry. The IP address right after via is your default gateway.
  • In VirtualBox NAT mode the gateway is usually 10.0.2.2 — a virtual router VirtualBox creates automatically for your VM.
  • The second line describes your local subnet. Your VM can reach any address in the 10.0.2.0/24 range directly, without going through the gateway.
  • If there is no default via line, your VM has no route to the internet. That means ping 8.8.8.8 in Phase 3 will fail — a useful diagnostic distinction.
Phase 3 of 5

Test Internet Connectivity

Having an IP address and a default gateway does not guarantee your VM can actually reach the internet — a misconfigured adapter, a blocked ICMP rule, or a NAT problem could still prevent communication. The ping command sends small test packets to a remote host and reports whether they come back. Pinging 8.8.8.8 — Google's public DNS server — tests the full path from your VM through the gateway and out to the internet. By using a raw IP address rather than a hostname, you isolate connectivity from DNS: if this ping fails, the network itself is the problem, not name resolution.

Run this command 
ping -c 4 8.8.8.8

The -c 4 flag sends exactly 4 packets and then stops automatically. Without it, ping runs until you press Ctrl+C.

Sample output — yours will differ 
PING 8.8.8.8 (8.8.8.8) 56(84) bytes of data.
64 bytes from 8.8.8.8: icmp_seq=1 ttl=118 time=12.4 ms
64 bytes from 8.8.8.8: icmp_seq=2 ttl=118 time=11.9 ms
64 bytes from 8.8.8.8: icmp_seq=3 ttl=118 time=12.1 ms
64 bytes from 8.8.8.8: icmp_seq=4 ttl=118 time=12.0 ms

--- 8.8.8.8 ping statistics ---
4 packets transmitted, 4 received, 0% packet loss, time 3004ms  ← 0% loss means connected
rtt min/avg/max/mdev = 11.9/12.1/12.4/0.2 ms
What to look for
  • 0% packet loss in the statistics line means your VM can reach the internet. 100% loss means it cannot.
  • The time= value is the round-trip latency in milliseconds. Under ~100 ms is normal for a home or campus network.
  • If you see Network is unreachable: check that your VM's network adapter is set to NAT or Bridged in VirtualBox / UTM, and that the host machine itself has internet access.
  • Some campus or corporate networks block ICMP (the protocol ping uses). If ping fails but you believe you have connectivity, try curl -I https://example.com as an alternative test and note what you see.
Phase 4 of 5

Query DNS

Pinging an IP address verifies raw connectivity, but nearly all real-world network traffic uses hostnames like example.com rather than IP addresses. Before the OS can connect to a hostname, it must translate the name to an IP address using the Domain Name System (DNS). The dig command lets you query a DNS server directly and inspect the full response. If Phase 3 succeeded but this phase fails, you know your network path is fine and the DNS configuration is specifically the problem — a useful distinction that guides your troubleshooting.

Run this command 
dig example.com

Note: dig is part of the dnsutils package and may not be installed by default. If the command is not found, install it first:

sudo apt install dnsutils

Sample output — yours will differ 
; <<>> DiG 9.18.x <<>> example.com
;; QUESTION SECTION:
;example.com.                   IN      A

;; ANSWER SECTION:
example.com.     86400    IN    A    93.184.216.34  ← resolved IP address

;; Query time: 28 msec
;; SERVER: 127.0.0.53#53(127.0.0.53)  ← DNS server used (systemd-resolved)
;; WHEN: Sat May 03 10:15:00 UTC 2026
;; MSG SIZE  rcvd: 56
What to look for
  • The ANSWER SECTION is the key part. It shows the hostname and the IP address it resolved to (93.184.216.34 for example.com — a domain maintained by IANA for documentation purposes).
  • The number before IN A (e.g., 86400) is the TTL (time to live) in seconds — how long this answer may be cached. 86400 = 24 hours.
  • The SERVER line tells you which DNS server handled the query. On Ubuntu Server this is typically 127.0.0.53, the local stub resolver managed by systemd-resolved. It forwards queries to an upstream server configured by DHCP.
  • If there is no ANSWER SECTION or you see a SERVFAIL status, DNS is not working. Check systemctl status systemd-resolved or examine /etc/resolv.conf.
Phase 5 of 5

Inspect Listening Services

Your VM is not just a client making outbound requests — it is also a server that other machines may connect to. Network services advertise their availability by listening on a numbered port. SSH, for example, listens on port 22 so remote administrators can log in. The ss command (socket statistics) shows you which TCP services are currently listening for incoming connections. Before any server is exposed to a network, a sysadmin should know exactly what is listening and which process owns each socket. This is your first look at that picture. The ss command replaces the older netstat and is available by default on Ubuntu Server.

Run this command 
ss -tlnp

Flags: -t TCP only  |  -l listening sockets only  |  -n numeric ports (no service-name lookup)  |  -p show the owning process.

Tip: if the Process column appears empty, run sudo ss -tlnp — showing all process names requires elevated privileges.

Sample output — yours will differ 
State   Recv-Q  Send-Q  Local Address:Port  Peer Address:Port  Process
LISTEN  0       128     0.0.0.0:22          0.0.0.0:*          users:(("sshd",pid=823,fd=3))  ← SSH on all IPv4 addresses, port 22
LISTEN  0       128     [::]:22             [::]:*             users:(("sshd",pid=823,fd=4))  ← SSH on all IPv6 addresses, port 22
What to look for
  • Each row is one listening socket. The Local Address:Port column shows the port number after the colon.
  • You should see port 22 — that is SSH, which you enabled during the Ubuntu Server installation in Week 1.
  • 0.0.0.0:22 means the service accepts connections on all IPv4 addresses. [::]:22 is the same for IPv6.
  • The Process column names the program that owns the socket (e.g., sshd). Run with sudo if it shows nothing.
  • If you installed additional services in earlier weeks, they may appear here too. Note any port numbers you do not recognize — researching them is exactly what a real audit would involve.
  • Common ports to recognize: 22 (SSH), 53 (DNS), 80 (HTTP), 443 (HTTPS). In Week 7 you will install Nginx, and port 80 will appear in this list.
