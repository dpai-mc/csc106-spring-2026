Type: Extra Credit (optional)

Points: 5 (added on top of your base grade)

CSLO assessed: CSLO 2 — Configure a Linux operating system as a platform for web-based and other applications.

Why Network Troubleshooting Matters

A Linux server that cannot reach the network is, for most purposes, useless. It cannot serve web pages, receive SSH connections, download updates, or resolve domain names. Network connectivity is the lifeline that connects your carefully configured system to the rest of the world.

The tricky part is that network problems can look identical on the surface ("it's not working") while having very different root causes underneath. A missing default route, a misconfigured DNS server, a firewall rule blocking traffic, a downed interface: all of these produce the same symptom from a user's perspective. The difference between spending five minutes fixing the problem and spending two hours guessing is having a structured troubleshooting process.

This exercise asks you to build that process. You will work through a realistic scenario, document the diagnostic commands you would run at each step, and explain what you expect to see (or not see) at each stage. Think of it as writing a runbook: a document you could hand to someone else so they could diagnose the same problem methodically.

The Scenario

Situation: "Server Cannot Reach the Internet"

You are responsible for an Ubuntu Server VM that was working fine yesterday. This morning, you tried to run sudo apt update and it failed with connection timeout errors. You tried to open an SSH session from your host machine and that still works (you can log in), but from inside the server, nothing that requires internet access is functioning.

Here is what you know so far:

  • The VM is running and you can log in via the hypervisor console or SSH from the host.
  • apt update hangs and eventually times out.
  • A colleague on the same network says their machines can reach the internet just fine.
  • No one has made intentional configuration changes to this server recently.

Your job: systematically determine where the problem is and document every step of your diagnostic process.

This scenario is common in real system administration. Something that worked yesterday stops working today, and you need to isolate the cause without randomly changing settings and hoping for the best. A structured, layer-by-layer approach saves time and prevents you from accidentally making things worse.

Required Activities

To complete this extra credit, you need to do two things: read the reference materials listed below, then produce a written troubleshooting walkthrough document. Your document should address each diagnostic step described here.

Your Troubleshooting Walkthrough Must Include:

Work through the scenario above using a structured, bottom-up approach. For each step, document:

  • The command you would run
  • What you expect to see if things are working correctly
  • What you expect to see if this is where the problem is
  • What your next action would be based on the result

Here are the diagnostic steps your walkthrough should cover (in order):

Step 1: Check the Network Interface

Is the interface up? Does it have an IP address assigned? Use ip addr (or ip a) to inspect this.

Document what a healthy output looks like versus what you would see if the interface is down or has no address.

Step 2: Check the Default Route

Even if the interface has an IP, the system needs to know where to send traffic destined for the internet. Use ip route to check.

Document what the "default via" line should look like, and what it means if that line is missing.

Step 3: Test Basic Connectivity (Ping the Gateway)

Can you reach the gateway/router? Use ping with the gateway IP from Step 2.

Document what success looks like and what "Destination Host Unreachable" or 100% packet loss tells you.

Step 4: Test Internet Connectivity (Ping an External IP)

Can you reach something beyond your local network? Use ping 8.8.8.8 (Google's public DNS server).

Document what it means if the gateway ping works but this one does not.

Step 5: Test DNS Resolution

Can the system translate hostnames to IP addresses? Use dig example.com or nslookup example.com.

Document what a successful DNS response looks like, and what it means if ping 8.8.8.8 works but ping example.com does not.

Step 6: Check for Firewall Rules Blocking Outbound Traffic

Could local firewall rules be preventing outbound connections? Use sudo ufw status or sudo iptables -L -n to inspect.

Document what you would look for in the output and how a misconfigured rule could cause this problem.

Step 7: Check Listening Services (Optional but Useful Context)

Use ss -tlnp to verify what services are running. This helps confirm the system is otherwise healthy and narrows the problem to the network layer.

Tip: The key insight in network troubleshooting is working from the bottom up. Start with the physical/link layer (is the interface up?), move to the network layer (do I have an IP and a route?), then the transport layer (can packets get out and come back?), and finally the application layer (does DNS work? can I reach specific services?). If you skip layers, you might waste time investigating DNS when the real problem is a missing route.

You do not need to actually break your VM's network to complete this assignment. You are documenting the process you would follow, including the commands and their expected output at each stage. That said, if you want to experiment (for example, temporarily removing your default route with sudo ip route del default to see what happens), that is a great way to learn. Just make sure you know how to restore your configuration afterward.

Tip: If you do experiment on your VM, take a snapshot first (VirtualBox makes this easy). That way you can always roll back if something goes sideways.

Resources to Read

Before writing your walkthrough, read both of these free resources. They will give you the technical details and command examples you need.

  • DigitalOcean: "How To Troubleshoot Common Networking Issues on Linux" — A structured tutorial that walks through diagnosing network problems with command-line tools. This is your primary reference for the troubleshooting methodology.
  • Ubuntu Server Guide: Networking — Official Ubuntu documentation covering network configuration, interfaces, and diagnostics. Read the sections related to troubleshooting and network inspection.

Both resources are free and do not require an account to access. You will find specific command examples, sample output, and explanations of common failure modes in these guides.

You may also find it helpful to revisit the commands from the Week 6 Network Basics Check assignment (ip addr, ip route, ping, dig, ss -tlnp). Your troubleshooting walkthrough builds on those same commands but adds a diagnostic reasoning layer on top.

Submission Requirements

What to Submit

Submit a single document (PDF, DOCX, or plain text) containing your structured troubleshooting walkthrough. Your document should include:

  1. A brief introduction (2-3 sentences) stating the problem scenario in your own words.
  2. A step-by-step diagnostic process covering at minimum Steps 1 through 6 above. For each step, include:
    • The exact command you would run
    • Example output showing what "healthy" looks like
    • Example output (or a description) showing what a failure looks like at that step
    • A sentence or two explaining what you would conclude and what you would do next
  3. A short conclusion (3-5 sentences) summarizing how this bottom-up approach helps isolate network problems efficiently.

Formatting Expectations

  • Use clear headings or numbered steps so the document is easy to follow.
  • Include actual command syntax (formatted as code or in a monospace font).
  • Expected output can be paraphrased or shown as sample terminal output. You do not need to run every command live, but your examples should be realistic and accurate.
  • Aim for roughly 400-700 words total (not counting command output). Quality and clarity matter more than length.

Grading

This assignment is worth 5 extra credit points. You will earn full credit if your walkthrough:

  • Covers all six diagnostic steps with appropriate commands
  • Shows understanding of what each command reveals and why the order matters
  • Includes realistic expected output (healthy and failure cases)
  • Demonstrates clear reasoning about how to isolate the root cause

Partial credit is possible if some steps are incomplete or if the reasoning is unclear.

A note on this exercise: Network troubleshooting is one of those skills that feels abstract until you actually need it. Then it becomes the most practical thing you know. Even if you do not attempt this extra credit, the mental model of "start at the bottom layer and work up" is worth internalizing. You will use it in Week 7, in Mini-Project 2, and in any future role that involves managing Linux systems.
