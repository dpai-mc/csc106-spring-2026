What a Strong Post Looks Like

Example — Strong Post (~200 words)

Two practices I would follow on a production server are: avoiding direct root login, and assigning users to groups with the minimum permissions they actually need.

For the first practice, I would make sure PermitRootLogin no is set in /etc/ssh/sshd_config and that any administrative user is in the sudo group instead. The reason this matters is accountability — sudo logs every command with a timestamp and username in /var/log/auth.log. If something breaks, you can trace exactly who ran what. Logging in as root leaves no such trail.

For the second practice, I would create role-specific groups — for example, a webadmins group that owns the Nginx document root, rather than making every admin a member of sudo. This applies the principle of least privilege: a developer deploying HTML files doesn't need the ability to install packages or restart services.

I did lock myself out once during Week 1. I ran usermod -G developers jdoe without the -a flag, which replaced all supplementary groups and removed jdoe from sudo. I had to log in as my original admin user and re-add the group with usermod -aG sudo jdoe. That mistake stuck with me — I always use -aG now.

Why this earns full credit: Describes two specific practices with exact commands or configuration file paths. Explains the why behind each practice (audit trail, least privilege). Includes a real personal experience with a specific mistake and resolution. Word count exceeds 150. Technical content is accurate.

Example — Weak Post (~60 words)

I think security is really important on Linux. You should use strong passwords and not use the root account. Also make sure to give users only the permissions they need. I haven't had any problems on my VM. Security is a big topic and there is a lot to learn. Overall I think this week's material was interesting.

Why this loses points: Under 150 words. Mentions practices only in vague, general terms — no commands, no file paths, no specifics. Does not explain why any practice matters. Claims no personal experience without adding anything else to compensate. The last two sentences are filler with no technical content.

Example — Strong Reply (~80 words, replying to the strong post above)

Your point about usermod -G without -a is one I hadn't thought about — I've been running that command in scripts for Mini-Project 1 and never checked whether it was replacing the group list. I'm going to audit my script tonight. One thing I'd add to your practices: I also audit /etc/passwd for any accounts with UID 0 other than root using awk -F: '($3 == 0) {print $1}' /etc/passwd. Did you end up adding that kind of check to your Mini-Project 1 script?

Why this earns full credit: Directly engages with a specific point from the original post. Adds a new technical idea (UID 0 audit command) not in the original post. Ends with a genuine follow-up question that invites further discussion. Shows the student is reading and thinking, not just responding to sound polite.

Example — Weak Reply (~15 words)

Great post! I totally agree with everything you said. Security is really important.

Why this earns no credit: This is a social acknowledgment, not a substantive reply. It adds nothing — no new idea, no follow-up question, no alternative perspective, no connection to course content. The rubric requires replies that advance the conversation.
