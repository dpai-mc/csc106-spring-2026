What is Password Aging?

Password aging is a set of policies that control how long a password can be used before it must be changed, how soon after a change the user can change it again, and when the system should warn the user that their password is about to expire.

On Linux, these policies are stored as numeric fields in /etc/shadow alongside the password hash itself. Each user account has its own set of aging values, which means you can apply different policies to different accounts.

Why does this matter?

In a classroom setting, password aging might seem like unnecessary overhead. On a personal VM, you probably do not care if your password is six months old. But in any organization, password policies exist for real reasons:

  • Limiting exposure time: If a password is compromised (through a data breach, shoulder surfing, or a sticky note on a monitor), a maximum age policy ensures the compromised password stops working after a defined period.
  • Compliance requirements: Many security frameworks (PCI-DSS, HIPAA, SOC 2, NIST 800-53) require password rotation policies. System administrators need to know how to implement and verify these policies.
  • Account lifecycle management: Expiration dates on accounts ensure that temporary users (contractors, interns, seasonal staff) lose access automatically without someone having to remember to disable them.
  • Auditing and accountability: Password aging fields provide a record of when passwords were last changed, which is useful during security audits.

The chage command ("change age") is the standard tool for viewing and modifying these policies on Linux. It works the same way on Ubuntu, Red Hat, CentOS, Debian, and most other distributions.

Scenario: Northfield Community Health Clinic

The Situation

You are the part-time IT administrator for a small community health clinic. The clinic has 12 employees who access a shared Linux server for scheduling, patient intake forms, and internal documentation. Last month, an employee left the clinic but their account remained active for three weeks before anyone noticed. During that window, someone used the former employee's credentials to access the scheduling system remotely.

The clinic's compliance officer has asked you to implement the following password policies on the server:

  • Passwords must be changed at least every 90 days
  • Users must wait at least 7 days before changing their password again (prevents cycling through passwords to get back to a favorite)
  • Users should receive a warning 14 days before their password expires
  • Accounts for temporary staff (volunteers, interns) should have a hard expiration date
  • If a user does not log in within 30 days after their password expires, the account should be automatically disabled

Your job is to demonstrate that you can implement these policies on a test user account using chage. You will do this on your Ubuntu Server VM using a test account (not your primary admin account).

Important: Do not apply restrictive password aging to your own admin account on the VM. If you lock yourself out, you will need to boot into recovery mode to fix it. Use a test user account for this exercise. If you still have the testuser account from the Week 5 videos or the Cisco lab, that works perfectly.

Required Tasks

Complete all of the following on your Ubuntu Server VM. Document each step with screenshots and a brief explanation.

If you do not already have a test user on your VM, create one now. This is the account you will apply password aging policies to.

sudo useradd -m -s /bin/bash -c "Clinic Test User" clinicuser
sudo passwd clinicuser

Set a password you can remember. You will not actually need to log in as this user, but the account needs a password for the aging policies to be meaningful.

Before making changes, check the current password aging settings for your test user.

sudo chage -l clinicuser

You should see output similar to this:

Last password change                    : Apr 27, 2026
Password expires                        : never
Password inactive                       : never
Account expires                         : never
Minimum number of days between password change  : 0
Maximum number of days between password change  : 99999
Number of days of warning before password expires : 7

The default on Ubuntu is essentially "no policy." The maximum is set to 99999 days (about 274 years), which effectively means the password never expires. Take a screenshot of this output as your "before" state.

Implement the clinic's requirement that passwords expire after 90 days.

sudo chage -M 90 clinicuser

The -M flag sets the maximum number of days a password remains valid. After 90 days, the user will be forced to change their password at next login.

In your write-up, explain what happens from the user's perspective when their password expires. (Hint: they are not locked out immediately. They are prompted to set a new password at their next login.)

Prevent users from immediately changing their password back to a previous one by cycling through changes.

sudo chage -m 7 clinicuser

The -m (lowercase) flag sets the minimum number of days between password changes. The user cannot change their password again until 7 days have passed since the last change.

In your write-up, explain why this policy exists. What problem does it solve? (Think about what happens if a user is forced to change their password but immediately changes it back to the old one.)

Give users advance notice that their password is about to expire.

sudo chage -W 14 clinicuser

The -W flag sets how many days before expiration the user starts seeing warning messages at login. With this setting, starting 14 days before the password expires, the user will see a message like "Warning: your password will expire in X days" each time they log in.

Automatically disable the account if the user does not log in within 30 days after their password expires.

sudo chage -I 30 clinicuser

The -I flag sets the number of inactive days after password expiration before the account is locked. This is the clinic's safety net: if an employee leaves and their password expires, the account becomes completely unusable 30 days later without administrator intervention.

This is different from password expiration. An expired password can still be changed by the user at login. An inactive (locked) account requires an administrator to unlock it.

For the scenario's temporary staff requirement, set a hard expiration date on the account. Pick a date roughly 6 months from today.

sudo chage -E 2026-10-27 clinicuser

The -E flag sets an absolute expiration date for the account. After this date, the account is completely disabled regardless of password status. This is ideal for contractor or intern accounts where you know the end date in advance.

Use a date format of YYYY-MM-DD. Pick a date that makes sense for your scenario (roughly 6 months out is fine).

Run chage -l again to confirm all your changes took effect.

sudo chage -l clinicuser

Your output should now look something like this:

Last password change                    : Apr 27, 2026
Password expires                        : Jul 26, 2026
Password inactive                       : Aug 25, 2026
Account expires                         : Oct 27, 2026
Minimum number of days between password change  : 7
Maximum number of days between password change  : 90
Number of days of warning before password expires : 14

Take a screenshot of this output as your "after" state. Compare it with the "before" screenshot from Task 2.

Also examine what changed in /etc/shadow:

sudo grep "^clinicuser" /etc/shadow

In your write-up, identify which fields in the shadow entry correspond to each setting you changed. The readings explain the field positions.

Command Reference

This table summarizes the chage options used in this exercise. Refer to the manpage readings for the complete list.

Flag Purpose Example
-l List current aging settings for a user chage -l username
-M Set maximum days before password must be changed chage -M 90 username
-m Set minimum days between password changes chage -m 7 username
-W Set warning days before password expires chage -W 14 username
-I Set inactive days after expiration before account locks chage -I 30 username
-E Set absolute account expiration date (YYYY-MM-DD) chage -E 2026-10-27 username

A note on case sensitivity: Notice that -M (uppercase) and -m (lowercase) do completely different things. Uppercase M is maximum age; lowercase m is minimum age. This trips people up. Double-check which one you are using before pressing Enter.
