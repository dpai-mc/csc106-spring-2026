Server Configuration: Networking, Web, and SSH

CSC 106 – Introduction to Linux II  ·  Week 7, Lab 5  ·  45 points  ·  CSLO 2, CSLO 3

Scenario

You are a junior systems administrator preparing your organization's Ubuntu Server for its first deployment. The server needs a stable, predictable IP address (so clients can always find it), a web server to serve content, a firewall to block unauthorized access, a scheduled health-monitoring task, and secure SSH access using key-based authentication instead of passwords.

This is the configuration that every real server goes through before it goes into production. By the end of this two-part assignment, your VM will look and behave like a real server — one you built and secured yourself, from scratch.

By completing this assignment you will demonstrate that you can:
  • Inspect the current network configuration using ip addr, ip route, and Netplan YAML files.
  • Configure a static IP address on Ubuntu Server using Netplan.
  • Install Nginx, create a custom web page, and verify it serves correctly with curl.
  • Configure ufw to enforce a default-deny firewall that allows only SSH and HTTP.
  • Create a cron job that logs Nginx service health on a schedule.
  • Generate an SSH key pair and configure passwordless key-based authentication.
  • Explain why each of these configurations contributes to a secure, functional server.

Part A · Step 1 of 5

Inspect Your Current Network Configuration

Before you change any network settings, capture a snapshot of your current configuration. You need three pieces of information to write your Netplan file correctly: your interface name, your current IP and subnet, and your default gateway. Do not skip this step — copying an example with the wrong interface name or gateway is one of the most common mistakes in this assignment.

Step 1 — Find your interface name and current IP address 
ip addr

Look for the interface that is not lo. Its name — commonly enp0s3, ens3, or enp0s1 depending on your hypervisor — is what you must use in your Netplan YAML. Find the inet line under that interface and note the IP address and prefix (e.g., 192.168.1.47/24).

Sample output — yours will differ 
2: enp0s3: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 state UP
    link/ether 08:00:27:a5:b3:c1 brd ff:ff:ff:ff:ff:ff
    inet 192.168.1.47/24 brd 192.168.1.255 scope global dynamic enp0s3  ← dynamic = DHCP, this will change
    valid_lft 86342sec preferred_lft 86342sec
Step 2 — Find your default gateway 
ip route

Look for the line that starts with default via. The IP address immediately after via is your gateway — you will need this in the routes section of your Netplan file.

Sample output — yours will differ 
default via 192.168.1.1 dev enp0s3 proto dhcp src 192.168.1.47 metric 100  ← 192.168.1.1 is the gateway
192.168.1.0/24 dev enp0s3 proto kernel scope link src 192.168.1.47 metric 100
Step 3 — View the current Netplan configuration 
cat /etc/netplan/*.yaml

Note the filename shown when you run ls /etc/netplan/ — you will edit this file in the next step. Common names are 00-installer-config.yaml and 50-cloud-init.yaml.

Sample output — current DHCP config (what you'll replace) 
network:
  version: 2
  ethernets:
    enp0s3:
      dhcp4: true   ← this is what we're changing to a static IP
Part A · Step 2 of 5

Configure a Static IP Address with Netplan

Servers need a fixed, predictable IP address. DNS records, firewall rules, and every client that connects to your server depends on that address staying the same across reboots. You will disable DHCP and assign a static IP by editing your Netplan YAML file.

⚠️ YAML Is Whitespace-Sensitive — Use Spaces, Not Tabs

YAML requires spaces for indentation — tab characters cause a parse error. Use exactly two spaces per indentation level. If netplan apply reports "expected mapping," a tab is almost certainly the cause. In nano, you can check: if your cursor jumps by 8 positions when you press Tab, you used a tab. Delete it and type two spaces.

Also: your interface name must match what you found in Step 1. Do not copy the example's enp0s3 if yours is different.

Step 1 — Edit your Netplan configuration file 
sudo nano /etc/netplan/00-installer-config.yaml

Replace the filename with your actual Netplan file name from Step 1. Replace the entire contents with a static IP configuration using the values you recorded in Step 1. Use the template below as your starting point.

Static IP template — replace ALL values with your own 
network:
  version: 2
  ethernets:
    enp0s3:           ← replace with YOUR interface name from Step 1
      dhcp4: no       ← disables DHCP; we take control of the address
      addresses:
        - 192.168.1.100/24   ← pick an unused IP on your subnet, same prefix
      routes:
        - to: default
          via: 192.168.1.1   ← your gateway from Step 1
      nameservers:
        addresses:
          - 8.8.8.8
          - 8.8.4.4

Choosing your static IP: Pick an address on the same subnet as your current DHCP address but with a different last octet (e.g., if your DHCP address is 192.168.1.47, choose something like 192.168.1.100). Avoid addresses that look like they might be assigned by DHCP to other machines. The subnet prefix (e.g., /24) should match your current prefix exactly.

Step 2 — Apply the configuration 
sudo netplan apply

No output means success. If you see an error, go back to your YAML file and check indentation. A common error message for a tab character: Error in network definition: expected mapping.

Step 3 — Verify your static IP is active 
ip addr show

Find your interface in the output. The inet line should now show your chosen static IP. It should no longer say dynamic, and it should show valid_lft forever.

Step 4 — Verify connectivity still works 
ping -c 3 8.8.8.8
ping -c 3 example.com

The first ping tests that you can reach the internet. The second tests that DNS resolution works. If the first succeeds but the second fails, check your nameservers section. If both fail, verify your via (gateway) address.

Recovery: what to do if network connectivity breaks

If netplan apply kills your network connection and you are connected via SSH, do not panic — you still have access to your VM through the VirtualBox or UTM console window. Open that window, log in directly, and fix the YAML file. The most common causes of a broken configuration are:

  • Wrong interface name (copy-pasted from the example instead of using yours)
  • Wrong gateway address (double-check against Step 1's ip route output)
  • Tab characters instead of spaces in the YAML
  • A static IP that conflicts with another device on the network

To revert to DHCP temporarily, change dhcp4: no back to dhcp4: true, remove the addresses, routes, and nameservers blocks, and run sudo netplan apply again.

Part A · Step 3 of 5

Install Nginx and Create a Custom Web Page

Nginx is a high-performance web server. You will install it, verify it is running as a system service, and replace the default welcome page with a custom page that identifies your server. This is the first time your Ubuntu Server will serve actual web content to clients.

Step 1 — Update the package index and install Nginx 
sudo apt update && sudo apt install nginx -y

Always run apt update before installing packages to refresh the package index. The -y flag answers yes to all prompts automatically.

Step 2 — Verify Nginx is running and enabled 
systemctl status nginx

Look for Active: active (running) and enabled in the output. "Enabled" means Nginx will start automatically after a reboot (Ubuntu's Nginx package sets this by default). If it is not enabled, run sudo systemctl enable nginx.

Step 3 — Confirm Nginx is listening on port 80 
ss -tlnp | grep :80

You should see a line showing Nginx listening on 0.0.0.0:80. If you ran ss -tlnp in Week 6 and only saw SSH on port 22, this is the new entry that Nginx adds.

Step 4 — Create your custom index page 
sudo nano /var/www/html/index.html

Write a simple HTML page that clearly identifies this as your server. It does not need to be elaborate — the goal is to prove the server is serving your content, not the Nginx default page. Include your name and something specific to this assignment.

Minimal example — customize with your own content 
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <title>CSC 106 Server</title>
</head>
<body>
  <h1>Week 7 Server — Your Name Here</h1>
  <p>Configured with static IP, Nginx, ufw, and SSH key authentication.</p>
  <p>Server hostname: your-hostname-here</p>
</body>
</html>
Step 5 — Verify Nginx is serving your custom page 
curl localhost

The output should be your custom HTML — not the Nginx default page. If you still see the default page, make sure you saved the file as index.html (not index.html.txt or similar).

Step 6 — Check web content ownership and permissions 
ls -la /var/www/html/

Your index.html should be owned by root with permissions 644 (readable by everyone, including the www-data user that Nginx runs as). This is the default after creating the file with sudo nano and is usually fine without further changes.

What to do if curl shows the default Nginx page instead of yours
  • Run ls /var/www/html/ and verify your file is named exactly index.html
  • Open the file with cat /var/www/html/index.html to confirm it has your content
  • If there is both an index.html and an index.nginx-debian.html, Nginx prioritizes index.html — so your file should take effect
  • Try sudo nginx -t to check the Nginx configuration for syntax errors before reloading
  • If you made changes after Nginx was already serving, run sudo systemctl reload nginx to pick up the changes
Part A · Step 4 of 5

Configure the Firewall with ufw

A server exposed to a network needs a firewall. ufw (Uncomplicated Firewall) provides a simple interface to the underlying Linux firewall. You will set a default-deny policy for incoming traffic, then explicitly open only the two ports your server needs: SSH (port 22) and HTTP (port 80).

⚠️ Critical Order: Allow SSH BEFORE You Enable the Firewall

If you enable ufw with a default-deny policy before allowing SSH, and you are connected via SSH, you will immediately lose your connection and be locked out. The VM console (VirtualBox or UTM window) always provides a way back in — but you want to avoid that disruption.

Always follow this sequence: set defaults → allow SSH → allow HTTP → then and only then enable.

Step 1 — Check the current firewall status 
sudo ufw status

This will likely show Status: inactive. Good — we will configure rules before activating.

Step 2 — Set default policies 
sudo ufw default deny incoming
sudo ufw default allow outgoing

These two commands establish the base policy: block all incoming traffic by default, but allow the server to initiate outgoing connections (needed for apt updates, DNS queries, etc.).

Step 3 — Allow SSH (do this before enabling the firewall) 
sudo ufw allow ssh

This allows TCP traffic on port 22. It is equivalent to sudo ufw allow 22/tcp. The firewall is still not active yet — you are just preparing the rule set.

Step 4 — Allow HTTP for Nginx 
sudo ufw allow http

This allows TCP traffic on port 80. Equivalent to sudo ufw allow 80/tcp. You can also use the application profile name: sudo ufw allow 'Nginx HTTP'.

Step 5 — Enable the firewall 
sudo ufw enable

The system will warn you that this may disrupt existing SSH connections. Because you allowed SSH in Step 3, it is safe to type y and proceed. The firewall will also be enabled automatically at every reboot.

Step 6 — Verify the final firewall state 
sudo ufw status verbose
Expected output 
Status: active
Logging: on (low)
Default: deny (incoming), allow (outgoing), disabled (routed)
New profiles: skip

To                  Action      From
--                  ------      ----
22/tcp              ALLOW IN    Anywhere          ← SSH open
80/tcp              ALLOW IN    Anywhere          ← HTTP open
22/tcp (v6)         ALLOW IN    Anywhere (v6)
80/tcp (v6)         ALLOW IN    Anywhere (v6)
Step 7 — Confirm Nginx still works through the firewall 
curl localhost

Traffic from localhost (the machine itself) is not filtered by ufw by default, so this should still return your custom HTML. The firewall controls external traffic coming from other machines.

If you accidentally lock yourself out of SSH

Open the VirtualBox or UTM window for your VM and log in directly through the console — this always works even when the network firewall blocks SSH from outside. Then run:

  • sudo ufw allow ssh — to add the SSH rule
  • sudo ufw status verbose — to confirm the rule was added

SSH connections should resume within a few seconds. Remember: when configuring firewalls on remote servers where you have no console access, getting locked out can be catastrophic. On a VM with console access, it is recoverable — but treat it as a learning moment about why order matters.

Part A · Step 5 of 5

Set Up the Health-Monitoring Cron Job

The final piece of Part A is a scheduled task. You will create a cron job that runs every hour and writes a timestamped Nginx health check to a log file. This is a simple but real form of server monitoring — you build up a history of whether your web server was running at each check interval.

Step 1 — Open your personal crontab for editing 
crontab -e

If this is your first time, it will ask which editor to use. Choose 1 for nano unless you prefer another editor. Add the cron entry below on its own line at the bottom of the file, then save and exit.

Cron entry — add this line (choose one format) 
# Option A — compact one-liner (logs just "active" or "inactive"):
0 * * * * echo "$(date): $(systemctl is-active nginx)" >> ~/nginx-health.log

# Option B — more detailed (logs date and full Nginx status):
0 * * * * date >> ~/nginx-health.log && systemctl status nginx --no-pager >> ~/nginx-health.log 2>&1

Understanding the schedule: 0 * * * *

Cron uses five time fields before the command: minute hour day-of-month month day-of-week. The value 0 * * * * means "at minute 0 of every hour, every day." A * means "any value."

Use Crontab Guru to build and verify schedule expressions in plain English.

Step 2 — Verify the cron entry was saved 
crontab -l

You should see your new entry listed. If the file appears empty or the entry is missing, run crontab -e again.

Step 3 — Test the command manually (do not wait an hour) 
echo "$(date): $(systemctl is-active nginx)" >> ~/nginx-health.log
cat ~/nginx-health.log

Run the cron command directly in the terminal to verify it produces correct output. The log file should appear in your home directory with a timestamped entry.

Expected log entry 
Wed May 07 14:00:01 UTC 2026: active   ← date and "active" (or "inactive" if Nginx were down)
Testing with a per-minute schedule and checking cron logs

To verify cron actually fires the job (not just that the command works manually), you can temporarily change the schedule to every minute by replacing 0 * * * * with * * * * *. Wait 1–2 minutes, then check your log file. Restore the schedule to 0 * * * * before submitting.

To see what cron is doing behind the scenes:

  • grep CRON /var/log/syslog — shows cron execution records in the system log
  • journalctl -u cron — shows the cron service journal (newer systems)

If the log file is not being created, verify that the log path is writable by your user. Using ~/nginx-health.log (your home directory) is the safest choice when running from your personal crontab (without sudo). If you use sudo crontab -e (root's crontab), the path /var/log/nginx-health.log also works.

Part B · Step 1 of 3

Generate an SSH Key Pair

SSH key authentication uses a mathematically linked pair of keys. The private key stays on the machine you connect from — never share it with anyone. The public key goes on the server in the target user's ~/.ssh/authorized_keys file. When you connect, the server challenges your SSH client to prove it holds the private key that matches the stored public key. No password is ever sent over the network.

For this assignment, you have two options for where to generate and use your keys. Choose the one that fits your setup:

Option 1 — Host to VM (recommended if SSH is set up) Generate on your host machine (Mac or Windows), copy to VM

Simulates the real-world setup. Generate the key on your laptop, then copy the public key to the VM using ssh-copy-id. Requires that SSH from your host to the VM is already working.

Option 2 — Same VM, two users (simpler) Generate on the VM as one user, copy to a second user account

Generate the key as your primary user, then manually copy the public key to a second account (like one you created in Week 5). Does not require host-to-VM SSH. Demonstrates the same concepts.

Generate the key pair 
ssh-keygen -t ed25519 -C "yourname@csc106"

-t ed25519 specifies the Ed25519 algorithm (modern, fast, and secure). The -C flag adds a comment label to identify the key — replace the value in quotes with your own name or label. Accept the default file location when prompted (press Enter). For the passphrase, an empty passphrase is fine for this assignment; a passphrase adds extra protection if your private key file is ever stolen.

Verify the key files were created 
ls -la ~/.ssh/
Expected output 
drwx------ 2 student student 4096 May 7 14:10 .          ← .ssh dir must be 700 (drwx------)
drwxr-x--- 5 student student 4096 May 7 14:10 ..
-rw------- 1 student student  411 May 7 14:10 id_ed25519      ← private key: 600 (-rw-------)
-rw-r--r-- 1 student student  103 May 7 14:10 id_ed25519.pub  ← public key: 644 (shareable)

Private key vs. public key: The file ending in .pub is your public key — this is the one you copy to the server. The file without .pub is your private key — never share this file, never paste it anywhere, never copy it to the server.

Part B · Step 2 of 3

Configure Key-Based Authentication

You need to place your public key in the target account's ~/.ssh/authorized_keys file. SSH checks this file when a connection is made — if the public key matches the private key the connecting client presents, authentication succeeds without a password.

File and directory permissions are not optional. SSH silently ignores authorized_keys if the permissions are too permissive — it will fall back to password authentication without giving you a useful error message. Setting correct permissions is a required part of this step.

Option 1: Using ssh-copy-id (from host machine to VM) 
ssh-copy-id yourusername@your-vm-ip-address

ssh-copy-id copies your public key, creates the ~/.ssh/ directory if needed, and sets correct permissions automatically. It is the simplest approach when connecting from your host machine to the VM. After running it, proceed to Step B3 to verify.

Option 2: Manual setup (same VM, two users — run as your primary user with sudo) 
# 1. Create the .ssh directory for the target user
sudo mkdir -p /home/targetuser/.ssh

# 2. Copy your public key into their authorized_keys file
sudo cp ~/.ssh/id_ed25519.pub /home/targetuser/.ssh/authorized_keys

# 3. Fix ownership — the files must be owned by the target user
sudo chown -R targetuser:targetuser /home/targetuser/.ssh

# 4. Set required permissions
sudo chmod 700 /home/targetuser/.ssh
sudo chmod 600 /home/targetuser/.ssh/authorized_keys

Replace targetuser with the actual username of the second account on your VM (such as a user you created in Week 5).

Verify permissions on the target .ssh directory 
sudo ls -la /home/targetuser/.ssh/
Required permissions — must match exactly 
drwx------ 2 targetuser targetuser 4096 May 7 14:15 .            ← 700, owner = targetuser
drwxr-x--- 4 targetuser targetuser 4096 May 7 14:15 ..
-rw------- 1 targetuser targetuser  103 May 7 14:15 authorized_keys  ← 600, owner = targetuser
What ssh-copy-id does under the hood

When you run ssh-copy-id user@server, it is doing exactly what the manual Option 2 steps do — it reads your ~/.ssh/id_ed25519.pub file and appends its contents to ~/.ssh/authorized_keys on the remote system, creating the file and directory with correct permissions if they do not exist. Understanding the manual steps means you can set up key auth on any system, even ones where ssh-copy-id is not available.

Part B · Step 3 of 3

Verify Passwordless Login

Test that SSH key authentication is working by connecting to the target account. A successful connection skips the password prompt entirely (or asks only for your key's passphrase if you set one) and drops you directly into a shell.

Test the connection — Option 1 (host to VM) 
ssh yourusername@your-vm-ip-address
Test the connection — Option 2 (same VM, two users) 
ssh targetuser@localhost

You should be logged in without a password prompt. Verify with whoami to confirm you are in the target account, then exit to return.

🔎 If SSH Is Still Asking for a Password — Troubleshoot Here

SSH silently falls back to password auth when key auth fails. Use verbose mode to see what is happening: 

ssh -v targetuser@localhost

Look for lines like Offering public key and Authentications that can continue. If the key is offered but rejected, work through this checklist:

  • Is .ssh/ owned by the target user (not root)? Check with ls -la /home/targetuser/
  • Is .ssh/ exactly 700 (drwx------)?
  • Is authorized_keys exactly 600 (-rw-------)?
  • Did you copy the .pub file (public key), not the private key?
  • Is the public key content all on a single line in authorized_keys? (No line breaks mid-key)
What "permission too open" looks like and how to fix it

If you run ssh -v and see the key being offered but rejected, and you suspect permissions, you can reproduce the problem deliberately:

  • Set .ssh/ to 777: sudo chmod 777 /home/targetuser/.ssh
  • Try to connect — SSH will now silently ignore the key and ask for a password
  • Fix it: sudo chmod 700 /home/targetuser/.ssh
  • Try again — passwordless login resumes

This exercise from the instructor's demonstration helps you understand why the permissions requirement exists: if the .ssh directory or authorized_keys file is world-writable, an attacker could append their own public key and gain access. SSH protects you by refusing to use keys stored with insecure permissions.
