Self-Signed TLS for Nginx Extra Credit

CSC 106 – Introduction to Linux II  ·  Week 7  ·  5 extra credit points  ·  CSLO 2

What This Activity Adds

In Lab 5, you configured Nginx to serve a custom web page over HTTP (port 80). Every request and response between a client and that page travels in plain text — anyone on the same network can read it. TLS (Transport Layer Security) encrypts that traffic so only the client and server can read it. HTTPS — HTTP over TLS — is the standard for every web server on the internet today.

In this extra credit activity you will add a TLS certificate to your Nginx server, configure it to accept encrypted HTTPS connections on port 443, and update your firewall rules to allow that traffic. When you are done, your VM will be reachable at both http:// (plaintext) and https:// (encrypted).

⚠️ Prerequisites — Complete Lab 5 First
  • Nginx is installed, running, and serving your custom index.html from /var/www/html/
  • ufw is active with SSH (22) and HTTP (80) allowed and default deny incoming
  • Your static IP address is configured and connectivity is verified

Self-signed vs. CA-signed certificates: A Certificate Authority (CA) is a trusted third party that vouches for your certificate's identity — this is what browsers use to show a padlock for real websites. A self-signed certificate is signed by you, not a recognized CA. Browsers will show a security warning ("Your connection is not private"), which is expected and normal for a learning environment. The encryption itself is identical — the only difference is who is vouching for the server's identity. For a class VM that is not publicly accessible, a self-signed certificate is the correct choice.

What you are building: TLS requires two files — a certificate (the public identity document) and a private key (the cryptographic secret). You will generate both with openssl, then tell Nginx where to find them.

Read Before You Start
  • DigitalOcean: How To Create a Self-Signed SSL Certificate for Nginx on Ubuntu — digitalocean.com — Self-Signed SSL for Nginx. The primary hands-on reference for this activity. The steps below follow the same approach.
  • Nginx Documentation: Configuring HTTPS Servers — nginx.org — Configuring HTTPS Servers. Explains the Nginx directives you will use (ssl_certificate, ssl_certificate_key, etc.) and how they work.
Step 1 of 6

Generate the Self-Signed TLS Certificate

You will use openssl to generate both the private key and the certificate in a single command. The private key stays on the server and is never shared. The certificate is the public document that your server presents to clients to identify itself.

Generate the private key and certificate 
sudo openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
  -keyout /etc/ssl/private/nginx-selfsigned.key \
  -out /etc/ssl/certs/nginx-selfsigned.crt

What each flag does:

req -x509 — create a self-signed certificate (skip sending a request to a CA). -nodes — do not encrypt the private key file with a passphrase (Nginx needs to read it automatically on startup). -days 365 — the certificate is valid for one year. -newkey rsa:2048 — generate a new 2048-bit RSA key pair at the same time. -keyout — path for the private key. -out — path for the certificate.

After running the command, openssl will ask you a series of questions about your identity. The only field that has any technical effect for a self-signed certificate is Common Name — enter your server's IP address (the static IP you configured in Lab 5). The other fields (country, state, organization) can be left blank with a period or filled in with anything.

Interactive prompts — example entries 
Country Name (2 letter code) [AU]: US
State or Province Name (full name) [Some-State]: California
Locality Name (eg, city) []: Ukiah
Organization Name (eg, company) [Internet Widgits Pty Ltd]: CSC 106
Organizational Unit Name (eg, section) []: Student Lab
Common Name (e.g. server FQDN or YOUR name) []: 192.168.1.100  ← use YOUR static IP
Email Address []:
Verify the certificate and key files were created 
sudo ls -la /etc/ssl/private/nginx-selfsigned.key
sudo ls -la /etc/ssl/certs/nginx-selfsigned.crt

The private key should be readable only by root (-rw------- root root). The certificate can be world-readable (-rw-r--r-- root root) since it is a public document.

Optional: Generate Diffie-Hellman parameters for stronger key exchange

The DigitalOcean tutorial includes this step for stronger security. DH parameters are used during the TLS handshake to negotiate a shared secret. The default parameters built into OpenSSL are fine for a learning environment, but generating a dedicated DH parameter file is considered best practice:

sudo openssl dhparam -out /etc/nginx/dhparam.pem 2048

This takes 30 seconds to 2 minutes to complete (it is computationally intensive). If you generate the DH parameter file, you will reference it in ssl-params.conf in the next step with: ssl_dhparam /etc/nginx/dhparam.pem;

Step 2 of 6

Create Nginx SSL Configuration Snippets

Rather than repeating SSL configuration in every Nginx server block, you will create two reusable configuration snippets in /etc/nginx/snippets/. One points to your certificate files; the other defines SSL security settings. Server blocks can then include these snippets with a single line each, keeping configuration clean and maintainable.

Create the certificate paths snippet 
sudo nano /etc/nginx/snippets/self-signed.conf
Contents of /etc/nginx/snippets/self-signed.conf 
ssl_certificate /etc/ssl/certs/nginx-selfsigned.crt;      ← path to your certificate
ssl_certificate_key /etc/ssl/private/nginx-selfsigned.key; ← path to your private key
Create the SSL parameters snippet 
sudo nano /etc/nginx/snippets/ssl-params.conf
Contents of /etc/nginx/snippets/ssl-params.conf 
ssl_protocols TLSv1.2 TLSv1.3;         ← only modern, secure protocol versions
ssl_prefer_server_ciphers on;           ← server chooses the cipher suite
ssl_session_cache shared:SSL:10m;       ← cache negotiated sessions for 10 min
ssl_session_timeout 10m;
ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384;
add_header Strict-Transport-Security "max-age=63072000" always;  ← HSTS: tell browsers to always use HTTPS

What ssl_protocols does: TLSv1.2 and TLSv1.3 are the current secure versions of TLS. Older versions (TLSv1.0, TLSv1.1) have known vulnerabilities and should not be offered. Restricting to 1.2 and 1.3 is standard practice on production servers.

What HSTS does: The Strict-Transport-Security header tells browsers that this server should only ever be accessed over HTTPS. After a browser sees this header once, it will refuse to make HTTP connections to the same host for the period specified (max-age=63072000 is two years). For a class VM, this is mostly informational — a real server would not add HSTS without first ensuring HTTPS is permanently configured.

If you generated DH parameters in Step 1 — add this line to ssl-params.conf

Add the following line anywhere in ssl-params.conf:

ssl_dhparam /etc/nginx/dhparam.pem;

This tells Nginx to use your custom DH parameters during key exchange rather than the defaults. It is not required but is considered best practice for production servers.

Step 3 of 6

Configure Nginx to Serve HTTPS

Nginx uses server blocks to define how it handles requests. Your current Lab 5 configuration has one server block listening on port 80 (HTTP). You will add a second server block that listens on port 443 (HTTPS) and includes the SSL snippets you just created. Both server blocks will serve the same /var/www/html/ content.

Open the default Nginx site configuration for editing 
sudo nano /etc/nginx/sites-available/default

Add the HTTPS server block below the existing HTTP server block. You can leave the original HTTP block unchanged — you are adding a second block, not replacing the first. Your file should contain both blocks when you are done.

Add this HTTPS server block to the end of the file 
server {
    listen 443 ssl;                    ← listen on port 443 with SSL enabled
    listen [::]:443 ssl;               ← same for IPv6
    server_name 192.168.1.100;         ← replace with YOUR static IP address

    include snippets/self-signed.conf; ← loads certificate and key paths
    include snippets/ssl-params.conf;  ← loads SSL security settings

    root /var/www/html;                ← same web root as your HTTP server
    index index.html;

    location / {
        try_files $uri $uri/ =404;
    }
}

Your final /etc/nginx/sites-available/default should have two server blocks — the original one listening on port 80 (from your Lab 5 setup) and the new one above listening on port 443. Both serve the same content. You are not replacing the HTTP server; you are adding HTTPS alongside it.

Optional: Add an HTTP-to-HTTPS redirect

A common production configuration redirects all HTTP traffic to HTTPS automatically. To do this, replace the contents of your port 80 server block with a redirect instead of serving content directly:

server {
    listen 80;
    listen [::]:80;
    server_name 192.168.1.100;
    return 301 https://$host$request_uri;
}

With this in place, any request to http://your-ip/ will automatically redirect to https://your-ip/. This is not required for the extra credit assignment but is worth understanding as standard practice.

If you add the redirect, note that curl localhost from Lab 5 will now return a 301 response instead of your HTML. Use curl -k https://localhost or curl -kL http://localhost (follow redirects) to verify your page still loads.

Step 4 of 6

Test the Configuration and Reload Nginx

Always test the Nginx configuration for syntax errors before reloading. A syntax error in a live configuration can bring down your web server entirely. The nginx -t command checks the configuration without actually applying it — if it reports success, the reload is safe to run.

Step 1 — Test for configuration errors 
sudo nginx -t
Expected output if configuration is valid 
nginx: the configuration file /etc/nginx/nginx.conf syntax is ok  ← no syntax errors
nginx: configuration file /etc/nginx/nginx.conf test is successful  ← safe to reload
If nginx -t Reports an Error — Do Not Reload

Read the error message carefully — it will include the filename and line number where the problem was found. Common causes:

The include snippets/self-signed.conf; or include snippets/ssl-params.conf; lines cannot find the files — verify the snippet files exist with ls /etc/nginx/snippets/ and that their names match exactly. A missing semicolon at the end of any directive line. A missing closing brace } for a server block. Fix the error, save the file, and run nginx -t again before reloading.

Step 2 — Reload Nginx to apply the new configuration 
sudo systemctl reload nginx

reload applies the new configuration without interrupting active connections. It is safer than restart, which would terminate all active connections. Either works for a class VM.

Step 3 — Confirm Nginx is now listening on port 443 
ss -tlnp | grep nginx
Expected output — nginx now listening on both port 80 and 443 
LISTEN 0  511  0.0.0.0:80   0.0.0.0:*  users:(("nginx",pid=1257,fd=6))  ← HTTP still running
LISTEN 0  511  0.0.0.0:443  0.0.0.0:*  users:(("nginx",pid=1257,fd=7))  ← HTTPS now listening
LISTEN 0  511  [::]:80      [::]:*     users:(("nginx",pid=1257,fd=8))
LISTEN 0  511  [::]:443     [::]:*     users:(("nginx",pid=1257,fd=9))
Step 5 of 6

Open Port 443 in the Firewall

Nginx is now listening on port 443, but your Lab 5 firewall configuration only allows ports 22 (SSH) and 80 (HTTP). HTTPS traffic on port 443 is currently blocked by ufw. You need to explicitly allow it — the same way you allowed SSH and HTTP in Lab 5.

Allow HTTPS through the firewall 
sudo ufw allow https

This is equivalent to sudo ufw allow 443/tcp. You can also use the Nginx application profile: sudo ufw allow 'Nginx Full', which opens both port 80 and port 443 at once.

Verify the updated firewall rules 
sudo ufw status verbose
Expected output — three ports now allowed 
Status: active
Default: deny (incoming), allow (outgoing), disabled (routed)

To                  Action      From
--                  ------      ----
22/tcp              ALLOW IN    Anywhere          ← SSH (from Lab 5)
80/tcp              ALLOW IN    Anywhere          ← HTTP (from Lab 5)
443/tcp             ALLOW IN    Anywhere          ← HTTPS (newly added)
22/tcp (v6)         ALLOW IN    Anywhere (v6)
80/tcp (v6)         ALLOW IN    Anywhere (v6)
443/tcp (v6)        ALLOW IN    Anywhere (v6)
Step 6 of 6

Test and Verify the HTTPS Connection

With Nginx configured and the firewall updated, test that your server is actually serving encrypted HTTPS responses. You will use curl with special flags to verify the connection, since a self-signed certificate will cause standard verification to fail.

Test HTTPS from the command line — basic test 
curl -k https://localhost

The -k flag (or --insecure) tells curl to skip certificate verification. This is expected and correct for a self-signed certificate — curl would otherwise refuse to connect because the certificate is not signed by a recognized CA. The output should be your custom index.html content.

Test HTTPS with certificate details — verbose test 
curl -kv https://localhost 2>&1 | head -40

The -v flag enables verbose output, which shows the TLS handshake details — the certificate presented, the cipher suite negotiated, and the protocol version used. This output confirms encryption is actually taking place, not just that the server responded.

Expected verbose output (abbreviated) 
* Connected to localhost (127.0.0.1) port 443
* ALPN: curl offers h2,http/1.1
* TLSv1.3 (OUT), TLS handshake, Client hello (1):   ← TLS handshake begins
* TLSv1.3 (IN), TLS handshake, Server hello (2):
* TLSv1.3 (IN), TLS handshake, Certificate (11):    ← server presents certificate
* Server certificate:
*  subject: CN=192.168.1.100                         ← Common Name you entered in Step 1
*  issuer: CN=192.168.1.100                          ← same as subject: self-signed
* SSL connection using TLSv1.3 / TLS_AES_256_GCM_SHA384  ← cipher negotiated
* Server auth using Basic with user ''
> GET / HTTP/2
> Host: localhost
< HTTP/2 200
< content-type: text/html
...
<!DOCTYPE html>                                      ← your index.html content

Browser warning is expected: If you try to visit https://your-server-ip from a browser (for example, from your host machine if port forwarding is configured), the browser will show a "Your connection is not private" or "Certificate not secure" warning. This is correct behavior — the certificate is not trusted by any browser CA. You can click through the warning (usually "Advanced" → "Proceed anyway") to reach your page. The encryption is still working; the browser is just warning you that it cannot verify who issued the certificate.

Inspect the certificate itself with openssl

To view the full details of the certificate you generated — including the subject, issuer, validity dates, and the public key — run:

sudo openssl x509 -in /etc/ssl/certs/nginx-selfsigned.crt -text -noout

The output will show the certificate in human-readable form. Look for:

  • Subject: CN = your-ip — the Common Name you entered during generation
  • Issuer: CN = your-ip — same as subject, confirming it is self-signed
  • Validity: Not After : — the expiration date (one year from generation)
  • Public Key Algorithm: rsaEncryption and RSA Public-Key: (2048 bit)

Understanding this output is a good foundation for reading any TLS certificate — the same fields appear in CA-signed certificates from real CAs like Let's Encrypt or DigiCert.
