1. Static IP Configuration with Netplan

Back in Week 1, your VM got its IP address from DHCP. That was fine for getting started, but servers need a fixed, predictable address. If the IP changes after a reboot, anything pointing to that address (DNS records, firewall rules, bookmarks) breaks.

What is Netplan?

Netplan is Ubuntu's network configuration system. It reads YAML files from /etc/netplan/ and translates them into instructions for the underlying network manager (usually systemd-networkd on Ubuntu Server). You describe what you want in YAML, and Netplan makes it happen.

Finding your current config

cat /etc/netplan/*.yaml

You will likely see a file named something like 00-installer-config.yaml or 50-cloud-init.yaml. Right now it probably says dhcp4: true.

Writing a static IP configuration

Here is what a typical static IP Netplan file looks like:

network:
  version: 2
  ethernets:
    enp0s3:
      dhcp4: no
      addresses:
        - 192.168.1.100/24
      routes:
        - to: default
          via: 192.168.1.1
      nameservers:
        addresses:
          - 8.8.8.8
          - 8.8.4.4

Key fields to understand:

  • dhcp4: no disables automatic address assignment
  • addresses is your chosen static IP with the subnet mask in CIDR notation
  • routes tells the system where to send traffic destined for the internet (your gateway)
  • nameservers tells the system where to look up domain names
YAML is whitespace-sensitive

Indentation matters. Use exactly two spaces per level. Never use tabs. A single misplaced space will cause netplan apply to fail with a cryptic error. Double-check your file before applying.

Check your interface name first

The interface name (enp0s3, ens3, enp0s1, etc.) varies by hypervisor. Run ip addr and look for the interface that currently has an IP address. Use that exact name in your YAML. Copying an example without checking will not work.

Applying and verifying

sudo netplan apply
ip addr
ip route
ping 8.8.8.8
ping example.com

If all four verification commands produce expected output, your static IP is working. If netplan apply breaks connectivity, you can still access the VM through the VirtualBox or UTM console window (not SSH) to fix the YAML file.

Deeper dive: What happens behind the scenes with Netplan

Netplan itself does not manage the network directly. It is a configuration abstraction layer. When you run sudo netplan apply, Netplan reads your YAML files, generates configuration for the backend renderer (on Ubuntu Server, that is typically systemd-networkd), and tells the backend to apply the new settings.

You can see what Netplan generates with sudo netplan generate, which writes backend config files without applying them. The generated files land in /run/systemd/network/. This is useful for debugging if you want to see exactly what Netplan is telling systemd-networkd to do.

If you ever need to troubleshoot further, networkctl status shows the state of all interfaces as systemd-networkd sees them.

2. Installing and Configuring Nginx

Nginx (pronounced "engine-x") is one of the most widely used web servers on the internet. It is lightweight, fast, and straightforward to configure for basic use. This is your first time setting up a network-facing service from scratch.

Installation

sudo apt update
sudo apt install nginx

Verify it is running

systemctl status nginx

You should see "active (running)." If you recall Week 3's systemd work, this is the same pattern: a service managed by systemd, started automatically after installation.

Enable Nginx to start at boot

sudo systemctl enable nginx

This creates a symlink so that Nginx starts automatically whenever the server boots. Remember from Week 3: enable means "start at boot," start means "start right now."

Create a custom web page

The default web root is /var/www/html/. Nginx ships with a default page, but you want to serve your own content to prove the server is working with your configuration.

sudo nano /var/www/html/index.html

Write something simple:

<!DOCTYPE html>
<html>
<head><title>My Server</title></head>
<body>
  <h1>CSC 106 Week 7 - Server is running!</h1>
  <p>Configured by [your name] on [date].</p>
</body>
</html>

Test locally

curl localhost

You should see your HTML content printed to the terminal. If you see the default Nginx page instead, make sure you replaced (or removed) the default file.

File ownership and permissions

Nginx runs as the www-data user. Files in /var/www/html/ need to be readable by that user. The defaults after installation are usually correct (root:root ownership, 644 for files, 755 for directories), but verify with:

ls -la /var/www/html/
Deeper dive: Where Nginx configuration lives

The main configuration file is /etc/nginx/nginx.conf. It sets global settings like the worker process count and includes other config files.

Site-specific configurations live in /etc/nginx/sites-available/. Active sites are symlinked into /etc/nginx/sites-enabled/. The default site configuration is at /etc/nginx/sites-available/default.

For this assignment, you do not need to modify these files. The default server block already serves from /var/www/html/. But knowing where to look is important for troubleshooting and for Mini-Project 2 if you want to customize further.

Before reloading Nginx after any config change, always test the syntax first:

sudo nginx -t

This catches errors before they break a running server.

How a web request reaches your custom page: client connects on port 80, ufw allows it, Nginx serves the file.

3. Firewall Configuration with ufw

A server without a firewall is like a house with every door and window open. Even on a class VM behind NAT, configuring a firewall is standard practice and builds habits you will need in any professional environment.

ufw (Uncomplicated Firewall) is Ubuntu's friendly interface to the underlying packet filtering system. It makes firewall rules readable and manageable.

The configuration sequence

Order matters here. Follow these steps in this exact order:

  1. Set default policies (deny incoming, allow outgoing)
  2. Allow SSH (so you do not lock yourself out)
  3. Allow HTTP (for Nginx)
  4. Enable the firewall
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow ssh
sudo ufw allow http
sudo ufw enable
sudo ufw status verbose
Always allow SSH before enabling ufw

If you enable the firewall without allowing SSH and you are connected via SSH, you will be immediately disconnected with no way back in through SSH. On a VM you can recover through the console window, but this is a mistake worth avoiding. Every sysadmin has done this at least once. Learn from their pain.

After enabling, sudo ufw status verbose should show something like:

Status: active
Default: deny (incoming), allow (outgoing), disabled (routed)

To                         Action      From
--                         ------      ----
22/tcp                     ALLOW IN    Anywhere
80/tcp                     ALLOW IN    Anywhere
Deeper dive: ufw application profiles

Instead of port numbers, ufw can use application profiles. Run sudo ufw app list to see what is available. After installing Nginx, you will see profiles like "Nginx HTTP", "Nginx HTTPS", and "Nginx Full".

You can use these instead of port numbers:

sudo ufw allow 'Nginx HTTP'
sudo ufw allow 'OpenSSH'

Both approaches (port numbers and app profiles) produce the same result. Use whichever feels clearer to you.

4. Scheduled Tasks with cron

Your server is running Nginx, but how do you know it is still running an hour from now? A day from now? Cron lets you schedule commands to run automatically at specified intervals, which is perfect for health checks, log rotation, and routine maintenance.

Cron schedule syntax

A cron entry has five time fields followed by the command to run:

minute  hour  day-of-month  month  day-of-week  command
  0      *         *           *        *         /path/to/script.sh

The example above runs at minute 0 of every hour, every day.

The assignment requirement

You need a cron job that writes the current date and Nginx status to a log file every hour. Here is one way to do it:

crontab -e

Then add a line like:

0 * * * * echo "$(date): $(systemctl is-active nginx)" >> /var/log/nginx-health.log

Verifying

crontab -l

This shows your saved crontab entries. To test without waiting an hour, temporarily change 0 * to * * (every minute), wait a minute, check the log file, then change it back.

Tip: Crontab Guru

If you are unsure whether your schedule expression does what you think, paste it into Crontab Guru (crontab.guru) to see a plain-English description of when it will run. It is a quick sanity check.

Deeper dive: Cron environment gotchas

Commands run by cron execute in a minimal environment. Your ~/.bashrc is not sourced, and the PATH is limited. This means commands that work fine in your interactive shell might fail in cron because cron cannot find them.

Two solutions:

  • Use full paths to commands: /usr/bin/systemctl instead of just systemctl
  • Set the PATH at the top of your crontab: PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin

If your cron job is not producing output and you expected it to, check /var/log/syslog for CRON entries or run journalctl -u cron to see if cron attempted to execute the job.

5. SSH Key Authentication

Password-based SSH login works, but it has a fundamental weakness: passwords can be guessed. SSH key authentication uses cryptographic key pairs instead, which are practically impossible to brute-force. This is standard practice on every production server.

How it works (conceptual)

  • You generate a key pair: a private key (stays on the client, never shared) and a public key (placed on the server)
  • When you connect, the server challenges the client to prove it holds the matching private key
  • No password crosses the network

Generating a key pair

ssh-keygen -t ed25519

Accept the default location (~/.ssh/id_ed25519). You can set a passphrase for extra protection or leave it empty for convenience.

Copying the public key to the server

If connecting from your host machine to the VM:

ssh-copy-id username@server-ip

If working entirely within the VM (simulating with a second user account), manually copy the key:

mkdir -p ~/.ssh
chmod 700 ~/.ssh
cat /path/to/id_ed25519.pub >> ~/.ssh/authorized_keys
chmod 600 ~/.ssh/authorized_keys

Verify passwordless login

ssh username@server-ip

It should connect without asking for a password (unless you set a key passphrase, in which case it asks for that instead).

Permissions are critical

SSH is strict about file permissions. If they are too open, SSH silently refuses to use the key and falls back to password auth. Required permissions:

  • ~/.ssh/ directory: 700
  • ~/.ssh/authorized_keys: 600
  • Private key file: 600

If key auth is not working, check permissions first. This is the most common cause.

Deeper dive: ed25519 vs RSA keys

ed25519 is a modern elliptic-curve algorithm. Keys are shorter, faster to generate, and considered more secure than RSA at equivalent security levels. It is the recommended choice for new keys.

If you encounter systems that do not support ed25519 (very old servers), RSA with 4096 bits is the fallback:

ssh-keygen -t rsa -b 4096

For this course, ed25519 is the right choice. You may see RSA keys in older tutorials and documentation.

6. Putting It All Together: A Functional Server

Step back for a moment and look at what you have built. After completing the five topics above, your server:

  • Has a predictable, static IP address (Netplan) so clients can always find it
  • Serves a custom web page (Nginx) to anyone who connects on port 80
  • Protects itself with a firewall (ufw) that allows only SSH and HTTP
  • Monitors its own health on a schedule (cron) without human intervention
  • Accepts secure, passwordless SSH connections (SSH keys) that cannot be brute-forced

This is not a toy setup. This is genuinely how small production servers are configured. The specific services might differ (maybe Apache instead of Nginx, maybe additional ports for a database), but the pattern is the same: assign a static address, install services, restrict access with a firewall, automate monitoring, and secure remote access.

Mini-Case: GreenLeaf Community Garden

Scenario

GreenLeaf Community Garden is a small nonprofit that wants to host a simple website showing their garden schedule, volunteer sign-up info, and photo gallery. They have one donated server (an old desktop running Ubuntu Server) and a volunteer sysadmin (that is you). The server will sit on their office network with a static IP of 10.0.1.50, gateway 10.0.1.1, and they use Google DNS (8.8.8.8).

Their requirements:

  • A static IP so the router's port forwarding rules always point to the right machine
  • A web server to host their HTML pages
  • A firewall that only allows web traffic and SSH for remote management
  • An hourly health check that logs whether the web server is running
  • SSH key login for the two volunteer admins (no passwords over the network)

Your task (think through these)

  1. What would the Netplan YAML look like for their network? (Interface: enp3s0, IP: 10.0.1.50/24, gateway: 10.0.1.1)
  2. After installing Nginx, what file would you edit to put up their custom homepage?
  3. What ufw commands would you run, and in what order?
  4. Write a cron entry that checks Nginx status every hour and appends to /var/log/greenleaf-health.log
  5. If volunteer admin "maria" needs SSH key access, what permissions must her ~/.ssh/ directory and authorized_keys file have?
Suggested answers for GreenLeaf scenario

1. Netplan YAML:

network:
  version: 2
