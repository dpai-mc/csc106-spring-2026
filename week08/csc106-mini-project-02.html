CSC 106 – Introduction to Linux II  ·  Week 8

Mini Project 2: Build and Document a Linux Server

80 Points  ·  CSLOs 1, 2, 3

Project Overview

This is your capstone project for the course. You will build a fully configured Ubuntu Server from scratch — installing the OS, configuring networking, setting up services, managing users, writing automation scripts, and hardening the system against common threats. Then you will document everything in a README and demonstrate the working server in a short screencast.

This project is cumulative. It draws on every skill you have practiced since Week 1: bash scripting, package management, user administration, systemd, Netplan, Nginx, ufw, cron, and SSH key authentication. You will not be writing code from memory — you will be applying concepts in a realistic, integrated context.

What you are building: A production-ready Linux server that a real administrator could hand off to a colleague, accompanied by documentation detailed enough that the colleague could reproduce your entire setup from scratch.

What You Will Demonstrate

  • Install Ubuntu Server with a non-root admin user and appropriate disk partitioning.
  • Install packages via a documented script or command sequence.
  • Configure the system to boot to multi-user.target with at least two services enabled.
  • Write at least one automation script (backup or health-check) and schedule it with cron.
  • Create multiple non-root users, manage groups, configure sudo, and set up a shared directory.
  • Assign a static IP via Netplan, configure ufw, and enable SSH key authentication.
  • Harden the system: disable root SSH login, open only needed ports, keep packages updated.
  • Produce a README and all configuration files that allow another student to reproduce your setup.
  • Record a 5–10 minute screencast demonstrating your working server.

Package Management & Installation Script
Automate your package installation — make your setup reproducible

Create a script that installs all the packages your server needs. Running this script on a fresh Ubuntu Server should bring it to a fully-equipped state without any manual package installation. This is the same concept as Mini-Project 1, but now integrated into a real working server rather than a standalone exercise.

Minimum Package Requirements

Your server must have at least the following installed:

  • nginx — web server (required for Phase 3)
  • ufw — firewall management (required for Phase 6)
  • curl — for testing HTTP connectivity
  • At least two additional packages of your choice that serve a purpose in your server setup
Task 2.1 — Write your package installation script

Create a script called install-packages.sh in your home directory. It must:

  • Run apt update and apt upgrade -y before installing anything
  • Install all required packages using an array and a loop (not individual apt install calls)
  • Check for errors after each installation step
  • Print clear status messages so you can see what is happening
nano ~/install-packages.sh
chmod +x ~/install-packages.sh
sudo ./install-packages.sh
Hint: Package array loop pattern

This pattern from Mini-Project 1 applies directly here. Define your packages as an array, then loop through them:

PACKAGES=("nginx" "ufw" "curl" "git" "htop")

for pkg in "${PACKAGES[@]}"; do
    echo "Installing $pkg..."
    apt install -y "$pkg" || { echo "Failed to install $pkg"; exit 1; }
done

The || { ...; exit 1; } pattern exits the script immediately if any package fails to install, rather than continuing with a broken state.

Task 2.2 — Verify installations

After running your script, confirm each package is installed:

dpkg -l nginx ufw curl
nginx -v
ufw version

Boot Configuration & Services
multi-user.target, Nginx, and SSH — a server that runs what it needs

A well-configured server boots to the right target and runs exactly the services it needs — no desktop environment, no unnecessary processes. Your server must boot to multi-user.target (the standard headless server mode) with at least two services enabled to start automatically.

Configure the Boot Target

Confirm your system is using multi-user.target:

systemctl get-default

If it shows graphical.target instead, switch it:

sudo systemctl set-default multi-user.target

Enable and Configure Nginx

Task 3.1 — Enable Nginx and SSH to start on boot
sudo systemctl enable nginx
sudo systemctl enable ssh
sudo systemctl start nginx

Verify both are active:

systemctl status nginx
systemctl status ssh
Task 3.2 — Create a custom Nginx page

Replace the default Nginx page with a custom one that identifies your server:

sudo nano /var/www/html/index.html

Create a simple HTML page that includes your name, the date, and a description of what this server does. Then verify it is served correctly:

curl http://localhost
Hint: Minimal custom index.html

Your custom page does not need to be elaborate — a few lines is fine. What matters is that it clearly identifies your server:

<!DOCTYPE html>

<head><title>My Project Server</title></head>

  <h1>CSC 106 Mini-Project 2 Server</h1>

Automation Script & Cron Job
Scheduled automation — the server maintains itself

Write at least one automation script and schedule it to run automatically with cron. Your script must do something genuinely useful: either a backup that archives important data, or a health-check that logs system state so you can review it later.

Choose Your Script Type

Option A — Backup script: Archives a target directory (such as /var/www/html or /etc/nginx) using tar, names the archive with a timestamp, stores it in a backup directory, and optionally removes archives older than a set number of days.

Option B — Health-check script: Logs disk usage, memory usage, and the status of key services to a dated log file in /var/log/. Useful for detecting problems before they become outages.
Task 4.1 — Write your automation script

Create your script in /usr/local/bin/ so it is available system-wide:

sudo nano /usr/local/bin/server-maintenance.sh
sudo chmod +x /usr/local/bin/server-maintenance.sh

Your script must include:

  • A shebang (#!/bin/bash)
  • Timestamped output so you know when it ran
  • At least one conditional (e.g., check whether the backup directory exists before writing to it)
  • Clear status messages or log entries
Hint: Backup script skeleton 
#!/bin/bash
# server-maintenance.sh — Backup /var/www/html daily

BACKUP_DIR="/var/backups/html"
SOURCE="/var/www/html"
TIMESTAMP=$(date +%Y%m%d_%H%M%S)
ARCHIVE="${BACKUP_DIR}/html_${TIMESTAMP}.tar.gz"

mkdir -p "$BACKUP_DIR"

echo "[$(date)] Starting backup of $SOURCE..."
tar -czf "$ARCHIVE" "$SOURCE" && echo "[$(date)] Backup saved: $ARCHIVE" \
  || echo "[$(date)] ERROR: backup failed"

# Remove archives older than 7 days
find "$BACKUP_DIR" -name "*.tar.gz" -mtime +7 -delete
echo "[$(date)] Cleanup complete."
Hint: Health-check script skeleton 
#!/bin/bash
# server-maintenance.sh — Daily health check

LOG="/var/log/health-check.log"
echo "=== Health Check: $(date) ===" >> "$LOG"
echo "--- Disk Usage ---" >> "$LOG"
df -h >> "$LOG"
echo "--- Memory ---" >> "$LOG"
free -h >> "$LOG"
echo "--- Service Status ---" >> "$LOG"
for svc in nginx ssh; do
    STATUS=$(systemctl is-active "$svc")
    echo "$svc: $STATUS" >> "$LOG"
done
echo "" >> "$LOG"
Task 4.2 — Test your script manually

Run it once by hand to confirm it works before scheduling it:

sudo /usr/local/bin/server-maintenance.sh

Then verify the output — check the backup archive or log file to confirm it produced the expected result.

Task 4.3 — Schedule with cron

Add a cron job to run your script daily. Edit root's crontab (since the script likely needs root access):

sudo crontab -e

Add a line that runs the script daily. Example — run every day at 2:00 AM:

0 2 * * * /usr/local/bin/server-maintenance.sh >> /var/log/maintenance.log 2>&1

Verify the cron entry:

sudo crontab -l
Hint: Cron field reference

Cron fields from left to right: minute hour day-of-month month day-of-week command

  • 0 2 * * * — 2:00 AM every day
  • */5 * * * * — every 5 minutes (useful for testing)
  • 0 0 * * 0 — midnight every Sunday

For testing, set the cron job to run every minute (* * * * *), wait a minute, confirm the output, then change it to your production schedule.

User Administration & Shared Directory
Multiple users, groups, sudo, and shared access

Real servers are not single-user environments. You will create multiple non-root accounts, organize them into a group, configure appropriate sudo access, and set up a shared directory that members of the group can read and write to.

Create Non-Root User Accounts

Task 5.1 — Create at least two non-root users

Create two new user accounts (use names of your choice — these are fictional team members):

sudo useradd -m -s /bin/bash alice
sudo passwd alice
sudo useradd -m -s /bin/bash bob
sudo passwd bob

The -m flag creates a home directory; -s /bin/bash sets the login shell.

Task 5.2 — Create a group and add users

Create a group called webteam and add both users:

sudo groupadd webteam
sudo usermod -aG webteam alice
sudo usermod -aG webteam bob

Verify group membership:

getent group webteam
Task 5.3 — Configure sudo access

Grant one of your new users limited sudo access. You have two options:

  • Option A: Add the user to the sudo group for full sudo access: sudo usermod -aG sudo alice
  • Option B: Use visudo to grant the user access to specific commands only

Document which approach you chose and why it fits your server's needs.

Task 5.4 — Create a shared directory

Create a shared directory that all webteam members can read and write:

sudo mkdir -p /srv/webteam
sudo chown root:webteam /srv/webteam
sudo chmod 2775 /srv/webteam

The 2775 permission sets the setgid bit, which ensures new files created in the directory inherit the webteam group rather than the creating user's primary group.

Task 5.5 — Verify shared directory access

Test that both users can create and read files in the shared directory:

sudo -u alice touch /srv/webteam/alice-file.txt
sudo -u bob touch /srv/webteam/bob-file.txt
ls -la /srv/webteam/

Both files should show webteam as the group owner.

Hint: Understanding the setgid bit (chmod 2775)

The 2 in 2775 sets the setgid bit on the directory. Without it, files created in /srv/webteam would inherit the creating user's primary group (likely their personal group), not webteam. The setgid bit forces new files to inherit the directory's group, making true sharing work correctly.

Breaking down 2775:

  • 2 — setgid bit
  • 7 — owner: read, write, execute
  • 7 — group: read, write, execute
  • 5 — others: read, execute (can list and enter, but not write)

Networking: Static IP, Firewall & SSH Keys
Stable, secure network access — the foundation of remote administration

This phase integrates the three major networking skills from Week 7: assigning a static IP so your server's address never changes, configuring ufw to allow only necessary traffic, and setting up SSH key authentication to eliminate password-based login.

Critical order: Always configure ufw to allow SSH before enabling the firewall. If you enable ufw without allowing port 22 first, you will lock yourself out of your SSH session. The sequence below is intentional — follow it exactly.

Step A: Static IP with Netplan

Task 6.1 — Identify your interface name and current IP
ip addr show
ip route

Note your interface name (commonly enp0s3 or ens33) and your current IP address. Choose an IP in the same subnet that is outside your DHCP range.

Task 6.2 — Edit your Netplan configuration
sudo nano /etc/netplan/00-installer-config.yaml

Replace DHCP configuration with a static assignment. Substitute your interface name, IP, gateway, and DNS:

network:
  version: 2
  ethernets:
    enp0s3:
      dhcp4: false
      addresses:
        - 192.168.1.100/24
      routes:
        - to: default
          via: 192.168.1.1
      nameservers:
        addresses: [8.8.8.8, 8.8.4.4]
YAML uses spaces — never tabs. Indentation errors in Netplan YAML will prevent the configuration from applying and can leave your network interface in an undefined state. If sudo netplan try shows errors, recheck your spacing carefully before confirming.
Task 6.3 — Apply and verify the static IP
sudo netplan try

If no errors appear after 120 seconds, confirm the change:

sudo netplan apply
ip addr show
ping -c 3 8.8.8.8

Step B: Firewall with ufw

Task 6.4 — Configure ufw (SSH first, then enable)
sudo ufw allow ssh
sudo ufw allow http
sudo ufw enable
sudo ufw status verbose

Confirm the output shows rules for ports 22 (SSH) and 80 (HTTP) before and after enabling.

Step C: SSH Key Authentication

Task 6.5 — Generate an SSH key pair and configure key login

Generate an ed25519 key pair (on whichever machine you are using to connect — host or VM):

ssh-keygen -t ed25519 -C "mp2-key"

Copy the public key to your server's admin user (using ssh-copy-id or the manual method):

ssh-copy-id -i ~/.ssh/id_ed25519.pub your-user@192.168.1.100

Test key-based login:

ssh your-user@192.168.1.100

You should connect without being prompted for a password.

Hint: Manual authorized_keys method (if ssh-copy-id is unavailable)

On the server, as your admin user:

mkdir -p ~/.ssh
chmod 700 ~/.ssh
nano ~/.ssh/authorized_keys

Paste the contents of your public key file (~/.ssh/id_ed25519.pub) into this file, then:

chmod 600 ~/.ssh/authorized_keys

Permissions must be exact: 700 on .ssh/ and 600 on authorized_keys. SSH will silently refuse key authentication if permissions are too loose.

Security Hardening
Defense in depth — close unnecessary doors, keep the system current

Security is not a single configuration change — it is a set of practices layered on top of each other. You will implement three baseline hardening measures that are standard practice on any internet-facing Linux server: disabling root SSH login, ensuring only necessary ports are open, and keeping packages updated.

Disable Root SSH Login

Task 7.1 — Disable root login via SSH

Edit the SSH daemon configuration:

sudo nano /etc/ssh/sshd_config

Find the PermitRootLogin line and set it to no. If it is commented out, uncomment it:

PermitRootLogin no

Reload the SSH service to apply the change:

sudo systemctl reload ssh
sudo sshd -T | grep permitrootlogin

The sshd -T command prints the effective configuration — confirm permitrootlogin no appears in the output.

Before disabling root login: Confirm that your admin user can log in via SSH with key authentication and can run sudo commands. If you disable root login while your admin account is misconfigured, you may lose remote access.

Firewall Audit

Task 7.2 — Verify only necessary ports are open

Check which ports are actually listening on your server:

ss -tlnp

Compare the listening ports against your ufw rules. Every listening port should have a corresponding ufw rule that intentionally allows it, or it should only be bound to localhost (127.0.0.1) and not reachable from the network.

sudo ufw status numbered
Hint: Interpreting ss -tlnp output

Each line shows a listening TCP socket. The Local Address:Port column tells you where it is listening:

  • 0.0.0.0:22 — SSH listening on all interfaces (expected)
  • 0.0.0.0:80 — Nginx listening on all interfaces (expected)
  • 127.0.0.1:631 — something on port 631 bound only to localhost (not reachable from network — generally safe)

If you see a port bound to 0.0.0.0 that you do not recognize and have not intentionally opened in ufw, investigate what service is listening there before proceeding.

Keep Packages Updated

Task 7.3 — Apply available security updates
sudo apt update
sudo apt upgrade -y

Document the state after upgrading. Note whether any packages were upgraded and whether a reboot is recommended (/var/run/reboot-required exists if a kernel update needs a reboot).

ls /var/run/reboot-required 2>/dev/null && echo "Reboot required" || echo "No reboot needed"

Documentation & Screencast
Write it up — make your work reproducible

The documentation and screencast together are worth as much as any individual technical phase. Graders will use your README to understand your decisions and verify that your configuration is reproducible. Your screencast demonstrates that everything actually works.

README Requirements

Your README must be detailed enough that a classmate who has completed the same labs could reproduce your server setup from scratch by following it. Include:

  • System overview — what your server does, what services run on it, and who the users are
  • Step-by-step setup instructions — every command needed to reproduce your configuration, in order
  • Configuration file explanations — explain the key decisions in your Netplan YAML, Nginx config, and sshd_config
  • Script documentation — what each script does, how to run it, and what output to expect
  • User and group structure — list all users, their groups, and their sudo access level
  • Security decisions — explain your ufw rules and hardening choices
  • Cron schedule — your full crontab with explanations of timing
  • Verification steps — commands that can be run to confirm each feature is working correctly
Write as you go: The easiest way to write a good README is to document each phase as you complete it, not all at once at the end. Copy the commands you actually ran, the output you actually saw, and the decisions you actually made. A README written from memory after the fact is always less accurate.

Screencast Requirements

Record a 5–10 minute video demonstrating your working server. Your screencast must show:

  1. Your server's hostname and IP address (hostname -I and ip addr)
  2. Nginx serving your custom page (curl http://localhost)
  3. Both services enabled and active (systemctl status nginx ssh)
  4. Your automation script running and producing output
  5. The shared directory with correct group permissions (ls -la /srv/webteam/)
  6. ufw status showing your firewall rules (sudo ufw status verbose)
  7. SSH key-based login working (connect without a password)
  8. Root SSH login disabled (sudo sshd -T | grep permitrootlogin)
Recording tool options: On Windows, use Xbox Game Bar (Win+G) or OBS Studio. On macOS, use QuickTime Player (File → New Screen Recording). Free options include OBS Studio and Loom. Upload to YouTube (unlisted is fine) or submit as a file to Canvas. Include the link or file in your submission.
Task 8.1 — Prepare a script for your screencast

Before recording, write out the commands you will run in order. This prevents stumbling during the recording and ensures you cover all required demonstrations. A dry run is worth the five minutes it takes.
