CSC 106 - Introduction to Linux II

Welcome to Week 4

You have reached the midpoint of the course, and this week is a turning point. For the past three weeks, you installed Ubuntu Server, explored the kernel and hardware, and learned to manage services with systemd. All of that was preparation for what you are doing now: writing a real script that automates the kind of setup work you have been doing by hand.

This entire week is organized around Mini-Project 1, a post-install configuration script. You will write a bash script that updates packages, installs software, creates a user, enables services, and logs everything it does. It is the largest single assignment so far (60 points), and it pulls together skills from every previous week.

The reading load is lighter this week on purpose. There is no new Cisco chapter to work through. Instead, most of your time should go toward planning, writing, testing, and debugging your script. First drafts rarely work perfectly, and that is completely normal. Debugging is part of the process, not a sign that something went wrong.

What You Will Learn This Week

By the end of this week, you should be able to:

  • Write a bash script that accepts command-line arguments and uses them to control script behavior.
  • Define and call functions within a bash script to organize code into reusable blocks.
  • Use arrays in bash to manage lists of data (packages to install, users to create, etc.).
  • Implement conditional logic and basic error handling to make scripts more robust.
  • Create a timestamped log of script actions for troubleshooting and documentation.
  • Automate post-installation configuration tasks (package installation, user creation, service management) in a single script.
  • Write a clear README that explains how to run a script and what it does.

CSLO alignment: This is the first week where all three CSLOs come together. Mini-Project 1 assesses CSLO 1 (Install and configure a Linux operating system), CSLO 2 (Configure a Linux operating system as a platform for web-based and other applications), and CSLO 3 (Describe Linux runlevels and startup scripts).

Mini-Project 1 Requirements at a Glance

Your script must include all of the following:

  • At least one command-line argument
  • Package index update and upgrade
  • Installation of 3 or more packages
  • Non-root user creation with sudo access
  • Enable and start at least one service
  • A timestamped action log
  • At least one function
  • At least one array
  • At least one conditional
  • Comments throughout the script
  • At least one error-handling check

Deliverables: your script file, a README explaining how to run it, and a screenshot or log of successful execution on your VM.

Start early. This project is not something you can do well in a single sitting the night before it is due. A good pace: plan your script by Monday or Tuesday, write and test through the middle of the week, and polish the README and log output by the weekend.

Key Concepts and Themes

Command-line arguments Positional parameters ($1, $#, $@), usage messages, and designing scripts that accept input at runtime.
Functions Breaking scripts into named, reusable blocks. Local variables, return codes, and the "main function" pattern.
Arrays Declaring arrays, looping through elements, and using them to manage lists of packages, users, or services.
Error handling Testing command success with if, the || operator, checking if users or files exist before acting, and running as root.
Logging Writing timestamped entries to a log file with tee, redirecting output, and building a record of what the script did.
Documentation Writing a README that someone else can follow. Comments in code. Treating documentation as part of the deliverable, not an afterthought.

A theme running through all of this: scripts should be readable, not just functional. Meaningful variable names, consistent comments, and logical structure matter. In professional environments, the person running your script is often not the person who wrote it.

Where This Week Fits

Weeks 1 through 3 formed a cluster focused on building and understanding your system:

  • Week 1 - Installed Ubuntu Server and learned package management with apt.
  • Week 2 - Explored the kernel, hardware layer, and the boot process.
  • Week 3 - Managed services and startup with systemd, explored runlevels and targets.

Week 4 (this week) is the pivot. You take everything from the first three weeks and put it to work in a script. This is the first of two mini-projects.

  • Week 5 - Users, groups, and system security. The scripting patterns you build this week (checking if a user exists, assigning groups) carry directly into Week 5.
  • Week 6 - File permissions, ownership, and networking fundamentals.
  • Week 7 - Network configuration, server setup, and security. That week's assignment involves cron jobs and service management, both of which benefit from this week's scripting skills.
  • Week 8 - Mini-Project 2 (capstone). Requires at least one automation script. Students who built a strong Mini-Project 1 will have patterns and templates to draw from.

In short, strong scripting habits this week make the rest of the course smoother.

Looking Ahead

Next week (Week 5) shifts focus to users, groups, and system security. You will learn how Linux manages user accounts, group memberships, and access control. Many of the patterns you practice this week (checking if a user exists, using useradd, assigning sudo access) show up again in Week 5, but from a system administration perspective rather than a scripting one.

The scripting skills you build now are not a one-week thing. They are tools you will keep using through the rest of this course and, honestly, through the rest of any work you do with Linux.
