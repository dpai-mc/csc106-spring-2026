CSC 106 – Introduction to Linux II  ·  Week 4

Mini Project 1: Post-Install Configuration Script

60 Points  ·  CSLOs 1, 2, 3

Scenario & Context

Your team manages infrastructure for a small nonprofit. Every few months, someone provisions a fresh Ubuntu Server VM — for a new project, a replacement after a failure, or a test environment. Each time, the same work follows: update the package index, install a handful of tools, create an admin account, and start the services the team relies on. Done by hand, this takes 30–45 minutes and is easy to get wrong.

Your job is to write a post-install configuration script that automates all of this. When a fresh Ubuntu Server VM comes online, your script should bring it to a fully configured state with a single command. This is exactly the kind of automation that Linux system administrators write and reuse — and it pulls together everything you have built since Week 1.

Why a Linux server specifically? Linux servers are free and open-source, run reliably without a GUI, are scriptable end-to-end with bash, and power the vast majority of server infrastructure worldwide. Automation on Linux is not an advanced skill — it is a baseline expectation.

What You Will Demonstrate

By completing this project you will show that you can:

  • Write a bash script that accepts command-line arguments and uses them to control behavior.
  • Define and call functions within a bash script to organize code into reusable blocks.
  • Use arrays in bash to manage lists of data such as packages to install.
  • Implement conditional logic and basic error handling to make scripts more robust.
  • Create a timestamped log of script actions for troubleshooting and documentation.
  • Automate post-installation tasks: package installation, user creation, and service management.
  • Write a clear README explaining how to run your script and what it does.

Plan Your Script
Topic 7 — Design before you code

Before writing a single line of bash, spend 10–15 minutes deciding what your script will do. Students who plan first write scripts that read like a coherent procedure; students who skip planning write scripts that are harder to debug and harder to grade.

Your script must accomplish all of the following:

  • Accept at least one command-line argument (a username for the admin account to create)
  • Update the package index and upgrade installed packages
  • Install at least 3 packages of your choosing
  • Create a non-root user account and add it to the sudo group
  • Enable and start at least one service
  • Write a timestamped log of all actions to a file
  • Include at least one function, one array, one conditional, and one error-handling check
  • Include comments throughout the script
Hint: Choosing packages and services

Pick packages that make sense together for a server environment. The course outline uses nginx, curl, git, and ufw as examples — but choose whatever fits your planned scenario. Three packages is the minimum.

For a service, any package that registers a systemd unit works. nginx and ssh (OpenSSH) are common choices. After installing a package, you can check for its service unit with systemctl list-units --type=service | grep nginx.

Script Structure: Shebang, Root Check & Arguments
Topics 1–2 — The foundation every administration script needs

Build this skeleton first and test it before adding any other logic. Starting with a solid foundation — the shebang, a root check, and argument validation — means you catch problems early when the script is still small.

The Shebang and Root Check

The shebang tells the OS which interpreter to use. The root check exits with a clear error message if the script is not run with sudo, which is required for apt and useradd.

#!/bin/bash
# post-install.sh — Automated post-install configuration script
# Usage: sudo ./post-install.sh <new-username>

if [ "$(id -u)" -ne 0 ]; then
    echo "Error: This script must be run as root. Use: sudo $0 <username>"
    exit 1
fi

Command-Line Arguments

Accept the admin username as $1. Validate that it was provided before proceeding. This is also where you print a usage message so the script is self-documenting.

if [ $# -lt 1 ]; then
    echo "Usage: sudo $0 <new-username>"
    exit 1
fi

USERNAME="$1"
echo "Configuration will create user: $USERNAME"

Functions and Arrays
Topics 3–4 — Organize your code; manage lists

Functions and arrays are what turn a script from a pile of commands into an organized, maintainable program. Add them to your script now, before you write the actual system tasks — this way the structure is in place before the complexity increases.

Defining Functions

Define all functions near the top of your script, then call them in order from a main() function at the bottom. This pattern makes the script's flow easy to read at a glance.

update_system() {
    echo "Updating package index and upgrading packages..."
    apt update -y
    apt upgrade -y
}

create_user() {
    local new_user="$1"    # local keeps this variable inside the function
    echo "Creating user: $new_user"
    useradd -m -s /bin/bash "$new_user"
    usermod -aG sudo "$new_user"
}

main() {
    update_system
    install_packages    # you will define this next
    create_user "$USERNAME"
    enable_services     # you will define this in Phase 5
}

main

Declaring and Iterating an Array

Use an array to hold your package list. This avoids calling apt install separately for each package and makes it easy to add or remove packages by editing one line.

# Declare at the top of your script, after the argument check
PACKAGES=("nginx" "curl" "git" "ufw")   # replace with your choices

install_packages() {
    echo "Installing ${#PACKAGES[@]} packages..."
    for pkg in "${PACKAGES[@]}"; do
        echo "  -> $pkg"
        apt install -y "$pkg"
    done
}
Hint: Array syntax pitfalls

Always quote array expansion: "${PACKAGES[@]}" not ${PACKAGES[@]}. Without quotes, package names containing spaces would be split incorrectly. It is a good habit even when your package names are simple.

${#PACKAGES[@]} gives the count of elements — useful for a log message like "Installing 4 packages...". ${PACKAGES[0]} accesses a single element by index (zero-based).

Creating a Non-Root User with Sudo Access

Your create_user() function creates the user from your $USERNAME argument and adds them to the sudo group. Add a check so the script handles the case where the user already exists — this makes the script safe to re-run.

create_user() {
    local user="$1"

    if id "$user" >/dev/null 2>&1; then
        echo "User '$user' already exists — skipping creation."
    else
        useradd -m -s /bin/bash "$user"
        usermod -aG sudo "$user"
        echo "User '$user' created and added to sudo group."
    fi
}

After running the function, verify from the terminal:

id testadmin
groups testadmin

To verify sudo access is correctly assigned: sudo -l -U testadmin. You should see a line containing (ALL : ALL) ALL.

Service Management, Error Handling & Logging
Topics 5–6 — CSLO 3

This phase connects directly to CSLO 3 — runlevels and startup scripts. You will enable a service so it starts at boot and starts it now, add error handling so failures are visible rather than silent, and create a timestamped log so there is a permanent record of what the script did.

Enabling and Starting a Service

systemctl enable configures the service to start at boot; systemctl start starts it in the current session. Your script must do both — this is the same distinction you practiced in Week 3.

SERVICES=("nginx")   # add to this array if you want to enable multiple services

enable_services() {
    for svc in "${SERVICES[@]}"; do
        echo "Enabling and starting $svc..."
        systemctl enable "$svc"
        systemctl start "$svc"
    done
}

Verify after running:

systemctl is-enabled nginx    # expected output: enabled
systemctl is-active nginx     # expected output: active
systemctl status nginx

Error Handling

Add at least one check that tests whether a command succeeded and exits (or logs a warning) if it did not. Either pattern below satisfies the requirement:

# Pattern 1: if/else block
if apt install -y nginx; then
    echo "nginx installed successfully."
else
    echo "ERROR: nginx installation failed." >&2
    exit 1
fi

# Pattern 2: inline with ||
systemctl start nginx || { echo "ERROR: failed to start nginx"; exit 1; }

Timestamped Logging

Define a log_action() function near the top of your script and call it at the start of each major step. The tee -a command prints to the terminal and appends to the log file simultaneously.

LOGFILE="/var/log/post-install-$(date +%Y%m%d-%H%M%S).log"

log_action() {
    local msg="$1"
    echo "$(date '+%Y-%m-%d %H:%M:%S') - $msg" | tee -a "$LOGFILE"
}

# Call it before each major step:
log_action "Starting post-install configuration for user: $USERNAME"
log_action "Updating package index..."
log_action "Installing packages: ${PACKAGES[*]}"

After running your script, view the log:

cat /var/log/post-install-*.log

Hint: Redirecting command output to the log file too

To also capture the output of commands like apt install into your log file, redirect their output:

apt install -y nginx >> "$LOGFILE" 2>&1

>> appends stdout to the log file; 2>&1 redirects stderr to the same place. This is optional but makes the log much more detailed for troubleshooting.

Topic 8 — Run the complete script; document it for others

Running the Complete Script

Debugging Techniques

If something does not work as expected, use these tools:

  • Trace mode: bash -x ./post-install.sh deployadmin — prints each command as it executes, prefixed with +. This is the fastest way to find where a script diverges from what you expect.
  • Exit on any error: Add set -e at the top of the script (after the shebang) to stop immediately when any command returns a non-zero exit code. Useful for finding silent failures.
  • Check exit codes manually: Run a command, then immediately run echo $? — 0 means success; anything else is an error.
Hint: Testing on a clean snapshot

If your VM hypervisor supports snapshots (VirtualBox does), taking a snapshot before your first full run lets you reset and re-test from a clean state. This simulates real-world conditions where a post-install script must be reliable on a truly fresh system, not one where you have already run half of the steps by hand.

Your user-existence check from Phase 4 is a small version of this idea — the script is safer to re-run because it checks before acting.
