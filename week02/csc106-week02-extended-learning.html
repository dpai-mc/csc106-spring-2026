Week 2: Additional Investigation and Extra Credit

Additional Investigation Activities

These activities extend the topics from the required assignment. They are designed to satisfy your curiosity and build practical intuition. You do not need to submit anything for these, but the skills and understanding you pick up here will help you in later weeks, especially when you work with systemd in Week 3 and scripting in Week 4.

Explore kernel modules in depth

What to try

In the required assignment, you ran lsmod and looked at the list of loaded modules. This investigation goes a step further.

  • Pick three modules from your lsmod output that you are curious about. Run modinfo <module_name> for each one and read through the output. Look at the description, the author, the license, and the "depends" field.
  • Try to find a module that depends on another module. You can spot this in the modinfo output or in the third column of lsmod (which shows how many other modules are using it). This gives you a feel for how modules form dependency chains, similar to how packages depend on other packages.
  • Look at the directory where modules live on disk: 
    ls /lib/modules/$(uname -r)/kernel/
    You will see subdirectories like drivers/, net/, fs/, and others. Browse around. This is where the actual .ko (kernel object) files are stored.
  • If you want to see what happens when a module is loaded, try watching dmesg output in real time. Open two terminal sessions (or use tmux if you have it). In one, run dmesg -w (the -w flag means "watch," so it keeps printing new messages). In the other, try loading a harmless module that is not already loaded. On many VMs, sudo modprobe dummy will load the dummy network interface module. Watch what dmesg reports. Then unload it with sudo modprobe -r dummy.
Why this matters: Kernel modules are how Linux supports such a wide range of hardware without a bloated kernel. When you troubleshoot a device that is not working (a network adapter, a storage controller), one of the first things to check is whether the right module is loaded. This investigation builds that diagnostic instinct.
Dig into /proc and /sys

What to try

You read about /proc and /sys in Linux Essentials Chapter 12 (and again in Chapter 13 during Week 1). Now explore them hands-on.

  • Read your kernel version from the filesystem instead of using uname: 
    cat /proc/version
    Compare this to the output of uname -a. They should match, because uname is just reading from /proc under the hood.
  • Check how long your system has been running: 
    cat /proc/uptime
    The first number is seconds since boot. Divide by 60 to get minutes, or by 3600 for hours. Compare this to the output of the uptime command.
  • Look at your kernel's command-line parameters: 
    cat /proc/cmdline
    This shows the options that GRUB passed to the kernel at boot. You will recognize some of these from your GRUB configuration examination in Part B of the required assignment.
  • Explore /sys/class/net/ to see your network interfaces: 
    ls /sys/class/net/
    You should see at least lo (loopback) and one or more network interfaces. Try cat /sys/class/net/<interface>/address to see the MAC address.
Why this matters: The /proc and /sys filesystems are how Linux exposes kernel and hardware information to user-space programs. Every monitoring tool, every system utility, every script that checks system status is ultimately reading from these virtual filesystems. When you cat /proc/cpuinfo, you are doing exactly what lscpu does internally (just with less formatting). Understanding this makes the "everything is a file" principle feel real.
Trace the boot timeline with systemd-analyze

What to try

In Part B of the required assignment, you looked at boot logs with journalctl -b. This investigation uses a different tool to see the boot process from systemd's perspective.

  • Run systemd-analyze by itself. It prints a summary of how long the boot took, broken into firmware, loader, kernel, and userspace time. On a VM, firmware and loader times may show as zero or very small, which is normal.
  • Run systemd-analyze blame to see which services took the longest to start. The output is sorted by time, slowest first. This is the command administrators use when they want to speed up boot times.
  • If you want a visual representation, try: 
    systemd-analyze plot > boot-timeline.svg
    This creates an SVG image showing every service and when it started relative to boot. You can transfer the file to your host machine and open it in a browser. It is a surprisingly detailed picture of your boot process.
Why this matters: This is a preview of Week 3, where you will work with systemd in depth. Seeing the boot timeline now, while the boot process concepts are fresh, helps you connect the abstract stages (firmware, bootloader, kernel, init) to concrete, measurable events on your own VM. When you get to Week 3 and start enabling and disabling services, you will already have a sense of how those services fit into the boot sequence.
Compare dmesg and journalctl output

What to try

The required assignment asks you to use both dmesg and journalctl -b. This investigation helps you understand the difference between them.

  • Run dmesg | wc -l to count the number of lines in the kernel ring buffer.
  • Run journalctl -k | wc -l to count kernel messages from the journal. The numbers may be similar but not identical, because journalctl -k pulls from systemd's journal (which is persistent across reboots if configured that way), while dmesg reads the kernel ring buffer (which is a fixed-size buffer in memory).
  • Run journalctl -b -1 to try viewing messages from the previous boot. If your system has only booted once since installation, this will say "No journal data found." If you have rebooted at least once, you will see the previous boot's messages. This is something dmesg cannot do, because the ring buffer only holds the current boot.
  • Try filtering journal output by priority: 
    journalctl -b -p err
    This shows only error-level messages from the current boot. On a healthy VM, you may see very few (or none). That is a good sign.
Why this matters: In practice, you will use both dmesg and journalctl regularly. Knowing when to reach for which tool saves time. Quick hardware question? dmesg. Need to look at a previous boot or filter by severity? journalctl. Building this habit early means you will not have to think about it later.

Extra Credit: GRUB Modification (5 points)

This is the one optional activity this week that earns points. It is included as an optional section within Lab 02 in Canvas. You complete it on your Ubuntu Server VM, just like the required assignment.

Before you start: Take a VM snapshot. Modifying GRUB configuration is safe when done correctly, but having a snapshot means you can always roll back if something goes wrong. This is a good habit for any configuration change on a Linux system.

What you will do

You will read about GRUB2 configuration from two free resources, then modify the GRUB timeout value on your VM and document the change. This is a natural extension of the required assignment, where you examined GRUB configuration with cat /etc/default/grub. Now you actually change it.

Step by step

1
Read the resources. Review the relevant sections of the GNU GRUB Manual and the Ubuntu Community Help Wiki page on Grub2 (both listed in the Resources section below). You do not need to read either document cover to cover. Focus on the sections about /etc/default/grub and the GRUB_TIMEOUT setting. The Ubuntu wiki page is particularly practical and Ubuntu-specific.
2
Check your current GRUB timeout. You already did this in Part B of the required assignment. Run: 
cat /etc/default/grub
Note the current value of GRUB_TIMEOUT. On a default Ubuntu Server install, this is typically 0 (meaning the GRUB menu is hidden and boot proceeds immediately). Take a screenshot of this output. This is your "before" screenshot.
3
Edit the GRUB defaults file. Open /etc/default/grub in a text editor with sudo: 
sudo nano /etc/default/grub
Change the GRUB_TIMEOUT value to something visible, like 10 (which means GRUB will display its menu for 10 seconds before auto-booting). If there is a line that says GRUB_TIMEOUT_STYLE=hidden, change it to GRUB_TIMEOUT_STYLE=menu so the menu actually appears. Save and exit.
4
Regenerate the GRUB configuration. The file you just edited is the source of truth, but GRUB does not read it directly. You need to regenerate the actual config file: 
sudo update-grub
You should see output indicating that a new grub.cfg was generated. If you see errors, double-check your edits to /etc/default/grub for typos.
5
Verify the change. Run cat /etc/default/grub again and take a screenshot showing the new timeout value. This is your "after" screenshot. If you want to fully verify, reboot your VM and watch for the GRUB menu. With a timeout of 10, you should see the menu appear and count down before booting. (On VirtualBox, you may need to watch the VM console window. On UTM, watch the display window.)
6
Document and submit. Your submission (as part of Lab 02) should include:
  • Your "before" screenshot showing the original GRUB_TIMEOUT value
  • Your "after" screenshot showing the modified value
  • A brief explanation (a few sentences) of what you changed, what command you used to apply the change, and what happened when you rebooted
CSLO alignment: This extra credit activity maps to CSLO 3 (Describe Linux runlevels and startup scripts). GRUB is the bootloader that starts the entire boot chain. Modifying its configuration is a direct, hands-on demonstration that you understand how the boot process is controlled, which is the foundation for the runlevel and startup script topics in Week 3.
Common mistakes to watch for
  • Forgetting to run sudo update-grub after editing. This is the most common mistake. If you edit /etc/default/grub but do not regenerate the config, nothing changes. GRUB reads /boot/grub/grub.cfg, not /etc/default/grub directly.
  • Editing /boot/grub/grub.cfg directly. Do not do this. That file is auto-generated. Any manual edits will be overwritten the next time update-grub runs (for example, after a kernel update). Always edit /etc/default/grub and then regenerate.
  • Leaving GRUB_TIMEOUT_STYLE=hidden while changing the timeout. If the style is set to "hidden," GRUB will not display the menu regardless of the timeout value. Change it to "menu" if you want to see the countdown.
  • Typos in the file. A missing equals sign or an extra space in the wrong place can cause update-grub to fail or produce unexpected behavior. Double-check your edits before saving.

These are the resources referenced by the extra credit activity and the additional investigation activities. All are freely available online.

Extra credit reading (required for the GRUB Modification)

  • GNU GRUB Manual The official manual for GRUB2. Focus on the "Simple configuration" section, which covers /etc/default/grub and its settings. You do not need to read the entire manual.
    GNU GRUB Manual
  • Ubuntu Community Help Wiki: Grub2 A practical, Ubuntu-focused guide to GRUB2 configuration. Covers common tasks like changing the timeout, setting the default boot entry, and troubleshooting. This is often easier to follow than the official GNU manual for specific Ubuntu tasks.

Practical Tips

Snapshots are your safety net

Before you try the extra credit (or any of the more adventurous investigation activities like loading and unloading kernel modules), take a VM snapshot. In VirtualBox, go to Machine > Take Snapshot. In UTM, right-click the VM and choose "Clone" or use the snapshot feature if available. If something breaks, you restore the snapshot and try again. Professional administrators do this before any significant change, and it is a habit worth building now.

Read before you type

For the extra credit, spend some time with the GNU GRUB Manual and the Ubuntu wiki page before you open a terminal. Understanding what GRUB_TIMEOUT, GRUB_TIMEOUT_STYLE, and update-grub do conceptually makes the actual modification straightforward. Students who jump straight to editing without reading tend to make small mistakes that take longer to fix than the reading would have taken.

Use the investigation activities to prepare for Week 3

The systemd-analyze investigation in particular is a direct preview of next week's material. If you try it now, you will already have a mental picture of how services start during boot when you begin working with systemctl in Week 3. That head start makes a real difference.

Time estimate

The additional investigation activities are open-ended. You could spend 20 minutes or 2 hours, depending on how deep you go. The extra credit (GRUB Modification) should take roughly 1 to 1.5 hours, including the reading. If you are planning to do both, budget about 2 to 3 hours total on top of the required assignment.
