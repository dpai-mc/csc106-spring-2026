Week 2: Hardware, the Kernel, and the Boot Process

CSC 106 - Introduction to Linux II

Welcome to Week 2. Last week you installed Ubuntu Server on a virtual machine, set up your first user, and started working with apt. That VM is now your working platform for the rest of the course, and this week you start looking under the hood at what makes it run.

The big question this week: what actually happens between pressing the power button and seeing a login prompt? To answer that, you will explore the Linux kernel, inspect your VM's hardware from the command line, learn about kernel modules, and trace the entire boot sequence from firmware through GRUB to systemd. This is the kind of foundational knowledge that makes everything else in Linux administration make more sense.

This week's focus: Understanding the layers underneath your running system. You will move from the kernel outward to hardware, then follow the boot process step by step. By the end of the week, you should be able to explain what your system is doing long before you ever see a login prompt.

Learning Goals

By the end of this week, you will be able to:

  • Describe the role of the Linux kernel and how it interacts with hardware.
  • Use commands to inspect hardware and kernel information (uname, lspci, lsusb, lsblk, lscpu, dmesg).
  • Explain what kernel modules are and how to list and load them.
  • Describe the Linux boot sequence from BIOS/UEFI through GRUB to the init system.
  • Identify the stages where problems can occur during boot.
CSLO alignment: This week maps to CSLO 1 (Install and configure a Linux operating system) and CSLO 3 (Describe Linux runlevels and startup scripts). Understanding the kernel and boot process is part of configuring a Linux system, and the boot sequence is the direct lead-in to runlevels and startup scripts that you will work with next week.

Key Concepts and Themes

As you work through this week's material, pay special attention to these ideas. They come up repeatedly throughout the rest of the course.

The kernel is the bridge

The kernel sits between your hardware and every program you run. It handles memory, processes, device access, and more. You do not interact with it directly most of the time, but everything you do on a Linux system depends on it. This week, commands like uname -a and lsmod give you a window into what the kernel is doing.

"Everything is a file"

Linux represents hardware devices as files in the /dev directory. Your disk might be /dev/sda or /dev/vda. The null device is /dev/null. This principle is one of the things that makes Linux distinctive, and it will keep showing up in later weeks when you work with permissions, networking, and server configuration.

The boot process is a chain

Booting is a sequence of handoffs: firmware to bootloader, bootloader to kernel, kernel to systemd. Each stage depends on the one before it. If any link in the chain breaks, the system does not come up. Understanding this chain is the foundation for troubleshooting boot problems, and it sets up next week's deep dive into systemd and services.

Kernel modules keep things flexible

Not everything is baked into the kernel. Drivers and features can be loaded as modules on demand, which keeps the kernel lean. On your VM, you will see modules for the virtual network adapter, the virtual disk controller, and filesystem drivers. Running lsmod and investigating a few modules with modinfo makes this concept concrete.

Looking Ahead: Week 3

Next week picks up exactly where this week leaves off. If Week 2 is about understanding how the system boots, Week 3 is about controlling what happens at boot. You will learn about runlevels and their modern replacement (systemd targets), use systemctl to manage services, change the default boot target, and perform clean shutdowns and reboots from the command line.

The boot sequence knowledge you build this week is essential for Week 3. When you use systemctl to enable a service or switch to rescue mode, you are working with Stage 4 of the boot process (the init system). Having a clear mental picture of how the system gets to that stage will make Week 3's material click much faster.

Course Arc

Where does this week fit in the course?

CSC 106 runs for eight weeks, and the content is organized into two major clusters.

Weeks 1 through 4: Build and automate

  • Week 1: Install Ubuntu Server, learn package management with apt.
  • Week 2 (this week): Understand the kernel, hardware, and boot process.
  • Week 3: Control services and boot behavior with systemd and runlevels.
  • Week 4: Advanced shell scripting, leading to Mini-Project 1 (a post-install configuration script).

Weeks 1 through 3 form a tight cluster: install the system, understand its internals, then manage its services. Week 4 layers scripting on top of that configured system. Everything from these four weeks feeds into Mini-Project 1.

Weeks 5 through 8: Secure and deploy

  • Week 5: Users, groups, and system security.
  • Week 6: File permissions, ownership, and networking fundamentals.
  • Week 7: Network configuration, web server setup, and security hardening.
  • Week 8: Integration and Mini-Project 2 (build and document a full Linux server).

The kernel and boot process knowledge from this week may feel conceptual right now, but it supports real troubleshooting and configuration decisions throughout the rest of the course. When something goes wrong with a service in Week 3 or a server in Week 7, understanding the layers underneath helps you figure out where the problem is.
