Scenario and Context

You have your Ubuntu Server up and running from Lab 1 - the nonprofit has a server, and it is connected to the internet. But before you start adding services and users, your manager asks a reasonable question: "Do you actually know what is inside that server?"

In a professional environment, understanding the hardware your software runs on is not optional. Mismatched drivers, unsupported hardware, and mysterious boot failures are real problems administrators deal with. More importantly, knowing how a Linux system starts up - from the moment the power button is pressed to the login prompt - is what separates someone who can recover a broken server from someone who has to reinstall it.

This week you will look under the hood. You will inspect the server's hardware using command-line tools, explore how the kernel manages devices through loadable modules, read the messages the system logged during boot, and trace the full boot sequence from firmware through GRUB to systemd. Everything you do here runs on the Ubuntu Server VM you built in Lab 1.

Note on virtual hardware: Your server is a virtual machine, so the "hardware" you see is emulated by the hypervisor. This is expected and normal. The commands work exactly the same way as on physical hardware, and the output is still informative - the hypervisor presents standard virtual devices to the guest OS, and those are what you will be inspecting.

How to submit command output in this lab: Because your Ubuntu Server runs without a graphical desktop, you cannot copy and paste terminal output.     ▶ Reading lsblk output

    The key columns:

    • NAME - device name (the full path is /dev/ + this name, e.g., /dev/sda)
    • SIZE - total size of the device or partition
    • TYPE - disk for a full drive, part for a partition
    • MOUNTPOINTS - where the partition is mounted in the filesystem (/, /boot/efi, etc.); blank means not currently mounted

    On VirtualBox VMs the disk is typically named sda. Some VMs using VirtIO storage show vda. The partition names append a number: sda1, sda2, etc.

    ▶ Why VMs show emulated hardware in lspci

    Hypervisors emulate standard, well-supported PCI devices so guest operating systems can use existing drivers without modification. Common virtual hardware you may see:

    • Intel 82540EM or Intel PRO/1000 MT - emulated network adapter (VirtualBox default)
    • VirtIO network or block devices - paravirtualized; faster than full emulation
    • PIIX4 or ICH9 - emulated chipset and storage controller
    • VirtualBox Graphics Adapter or VMware SVGA - virtual display adapter

    Seeing these entries means the kernel successfully detected and can communicate with the hardware the hypervisor is presenting - even though none of it is physically installed in your computer.

  6. 6

    Read early boot messages - dmesg

    The kernel maintains a ring buffer of messages logged since it started - hardware detection, driver loading, filesystem mounting, and ongoing hardware events. This buffer is what dmesg reads. Limiting output to the first 50 lines shows the earliest boot events: the very first things the kernel reported after it began running.

    $ sudo dmesg | head -50

    On some Ubuntu versions, reading the full ring buffer requires sudo. If the command runs without it, that is fine too - try without sudo first.

    Pick two specific lines visible in your screenshot and explain what each one is reporting. Include the timestamp (the number in brackets) when referencing a line. Look for lines about the kernel version, CPU detection, memory, or a device being recognized.

    ▶ Reading dmesg timestamps and filtering output

    Each line begins with a timestamp like [ 0.000000] - seconds elapsed since the kernel started. Lines at or near 0.000000 are the very first kernel actions: printing its version string, mapping memory, and detecting the CPU. Later timestamps show drivers initializing and devices coming online.

    Useful ways to search dmesg output:

    • sudo dmesg | grep -i cpu - CPU-related messages
    • sudo dmesg | grep -i "eth\|net" - network initialization
    • sudo dmesg | grep -i "error\|fail" - errors or failures

    You can also pipe the full output through less to scroll: sudo dmesg | less (press q to quit).

Boot Process and GRUB Configuration

In this part you will examine the bootloader configuration and review the full boot log from the current session. The goal is to connect the boot sequence you read about - firmware, GRUB, kernel initialization, systemd - to what actually happened on your own VM.

  1. 1

    Examine the GRUB configuration

    GRUB2 uses a two-file design: /etc/default/grub is the human-editable settings file, and /boot/grub/grub.cfg is the actual boot configuration - generated from it. Read the editable source:

    $ cat /etc/default/grub

    What options appear in GRUB_CMDLINE_LINUX_DEFAULT? These are kernel command-line parameters passed to the kernel at every boot. Explain what the options you see do. (If the value is "quiet splash", explain what each of those two options tells the kernel to do.)

    The comment at the top of /etc/default/grub says to run update-grub after making changes. This is because /boot/grub/grub.cfg is generated - you are not supposed to edit it directly. Why do you think GRUB uses this two-file design (a simple editable source and a generated config) instead of one file you edit directly?

    ▶ The /etc/default/grub and grub.cfg relationship

    /etc/default/grub is a short, readable settings file with a handful of variables. After you edit it, running sudo update-grub (or sudo grub-mkconfig -o /boot/grub/grub.cfg) regenerates the full /boot/grub/grub.cfg.

    /boot/grub/grub.cfg is a complex, automatically generated file. It contains an entry for every installed kernel version (current, older backups, recovery modes) and is re-created whenever a kernel is installed or removed. If you edited it directly, your changes would be silently overwritten the next time the kernel updates. The two-file design keeps human-editable settings separate from machine-generated output.

    ▶ Navigating the journalctl -b log

    The full boot log can be hundreds of lines. Useful ways to explore it:
