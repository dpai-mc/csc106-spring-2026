Lab 3: systemd Exploration

CSC 106 — Introduction to Linux II  •  Week 3

25 points +5 points extra credit CSLO 1 CSLO 2 (extra credit) CSLO 3

Scenario

You have just started as a junior Linux systems administrator at Summit Ridge Hosting, a small company that manages Ubuntu Server virtual machines for clients. On your first day, your supervisor hands you access to a freshly installed Ubuntu Server 24.04 VM and says:

"Before we put any client workloads on this server, I need you to document exactly what services are running at boot, confirm that SSH is healthy, and prove you know how to control the service manager. I also need to know you can get into single-user maintenance mode if something goes wrong at 3 AM."

This lab is that documentation task. You will explore systemd—the service and init manager running on your Ubuntu Server VM—and demonstrate that you can manage services and system targets from the command line.

Objectives

By completing this lab you will demonstrate the ability to:

  • Verify that systemd is running as PID 1 and check its version. (warm-up)
  • List all active services using systemctl and interpret the output. (Task a)
  • Check the detailed status of the ssh and cron services. (Task b)
  • Enable and disable a service, and verify the filesystem change in the .wants/ directory. (Task c)
  • Identify the current default systemd target and explain what it means. (Task d)
  • Change the default target to rescue.target, reboot into it, observe the result, and restore multi-user.target. (Task e)

Extra Credit (+5 pts): Set up a working cron job on your VM, verify it runs, and document it with screenshots and a written explanation of the schedule expression.

1

Meet systemd

Warm-up — ungraded

Before diving into service management, confirm that systemd is actually the init system running on your VM. This phase is ungraded; it orients you for the graded tasks ahead.

Step 1 — Confirm systemd is PID 1

Run the following command. PID 1 is the first user-space process the kernel starts after booting. On a systemd-based system, the command output will show systemd in the CMD column.

ps -p 1

Step 2 — Check the systemd version

This shows the version of systemd installed on your Ubuntu system, along with compile-time options.

systemctl --version
What to look for

The ps -p 1 output should show systemd or /sbin/init (a symlink to systemd on Ubuntu) in the CMD column. The systemctl --version output will start with a line such as systemd 255 followed by compile-time options in parentheses.

2

List Active Services

Task (a) — graded

Your supervisor wants a snapshot of everything that is currently running on the server. Use systemctl to list all active service units.

Step 1 — List all active services

Run this command in your terminal:

systemctl list-units --type=service --no-pager

The --no-pager flag sends the full output directly to the terminal instead of opening it in a scrollable viewer. If the list is long, scroll your terminal window up and take multiple screenshots to capture the full output including the summary line at the bottom.

Understanding the output columns
  • UNIT — the unit name, ending in .service
  • LOAD — whether systemd parsed the unit file (loaded is normal)
  • ACTIVE — the high-level state (active = currently running)
  • SUB — a more detailed state; for services, running is healthy
  • DESCRIPTION — a human-readable label from the unit file

The summary line at the bottom shows the total count of loaded, active, and running units.

Upload one or more screenshots if the output spans multiple screens. Include the summary line at the bottom if visible.

3

Check Service Status

Task (b) — graded

Two services that must be healthy on this server are ssh (remote login) and cron (scheduled tasks). Use systemctl status to check each one in detail.

Step 1 — Check the SSH service

On Ubuntu, the SSH daemon service is named ssh:

systemctl status ssh

Take a screenshot of the full output, including the Loaded and Active lines and the log entries at the bottom.

Reading systemctl status output
  • The first line shows the unit name and its description from the unit file.
  • Loaded: — path to the unit file and whether the service is enabled or disabled (controls boot behavior).
  • Active: — whether the service is running right now. A green dot and active (running) is healthy.
  • Main PID: — the process ID of the service's main process.
  • The last several lines show recent journal log entries for this service.

Key distinction: Enabled and active are independent. A service can be enabled (configured to start at boot) but inactive right now—or active right now but disabled (will not start on the next boot). You will see this clearly in Phase 4.

Capture the full block including Loaded, Active, Main PID, and the recent log lines.

Step 2 — Check the cron service

systemctl status cron

Take a screenshot of the full output.

4

Enable and Disable a Service

Task (c) — graded

Now you will enable and disable a service and observe what changes in the filesystem when you do. You will use ModemManager as the test subject—it manages cellular modems, which your server VM does not have, so toggling it has no effect on your running system.

💡 If ModemManager is not present

Run systemctl status ModemManager first. If the system says the unit is not found, use snapd instead. Substitute snapd for ModemManager in every command below. Both are safe to enable and disable for this exercise.

Step 1 — Check the initial state

First, check whether the service is currently enabled or disabled. Look for the word enabled or disabled on the Loaded line.

systemctl status ModemManager

Take a screenshot of this output.

Step 2 — Enable the service and inspect the result

Enable the service so it would start at the next boot, then look at what changed in the filesystem:

sudo systemctl enable ModemManager ls -l /etc/systemd/system/multi-user.target.wants/

Run both commands and take a screenshot of the combined output.

What enabling does under the hood

When you run systemctl enable, systemd reads the [Install] section of the unit file and finds a WantedBy=multi-user.target directive. It then creates a symbolic link:

/etc/systemd/system/multi-user.target.wants/ModemManager.service → /lib/systemd/system/ModemManager.service

That symlink is what makes the service start when the system reaches multi-user.target at boot. Disabling the service removes the symlink.

Both commands and their output should be visible. Upload two screenshots if needed.

Step 3 — Disable the service and verify

Disable the service and check whether the symlink was removed:

sudo systemctl disable ModemManager ls -l /etc/systemd/system/multi-user.target.wants/

Take a screenshot of both commands and their output.

Both commands and their output should be visible.

5

Identify the Default Target

Task (d) — graded

A systemd target is a named group of units that defines a system state, similar to the numbered runlevels used in older Linux systems. The default target determines what state the system boots into.

Step 1 — Check the current default target

systemctl get-default

Take a screenshot of the output.

Runlevels and targets — the mapping

Traditional SysV init used numbered runlevels. systemd replaces them with named targets:

SysV Runlevel systemd Target Description
0poweroff.targetSystem halt
1rescue.targetSingle-user / maintenance mode
3multi-user.targetMulti-user, network, no GUI
5graphical.targetMulti-user with desktop GUI
6reboot.targetSystem reboot

Ubuntu Server defaults to multi-user.target—the state you have been working in since Week 1: no desktop GUI, network services running, normal login prompt.

6

Switch to Rescue Mode and Back

Task (e) — graded

Your supervisor wants confirmation that you can enter rescue mode—the minimal single-user shell used for emergency system repairs. In this phase you will change the default target to rescue.target, reboot to observe it, and restore the system to normal operation.

⚠ Read all five steps before you start

When the system boots into rescue mode it drops to a minimal root shell. Your normal user login will not be available. You will change the default target back from inside that shell, then reboot again to confirm everything is restored. Follow the steps in order and do not skip ahead.

Step 1 — Confirm your starting point

Verify you are on multi-user.target before making any changes:

systemctl get-default

Take a screenshot showing multi-user.target.

Step 2 — Set rescue.target as the default and confirm

sudo systemctl set-default rescue.target systemctl get-default

Take a screenshot of both commands and their output together.

Step 3 — Reboot into rescue mode

Reboot the VM. The system will boot into rescue mode instead of the normal multi-user environment:

sudo reboot

After the reboot you will see a minimal root shell. It may look something like:

# You may be prompted for the root password, or dropped directly to a prompt [ OK ] Reached target rescue.target. root@hostname:~#

Take a screenshot of the rescue shell prompt and the boot messages above it.

Step 4 — Restore multi-user.target from inside rescue mode

From the rescue shell, set the default target back to normal, then reboot:

systemctl set-default multi-user.target reboot

Note: in rescue mode you are already operating as root, so sudo is not needed. Take a screenshot of both commands in the rescue shell.

Step 5 — Verify normal boot is restored

After the system reboots normally, log in and confirm the default target is back:

systemctl get-default

Take a screenshot showing multi-user.target.

What to do if you get stuck in rescue mode

If rescue mode prompts you for a root password and you have not set one, try pressing Enter. If that does not work, you can force a recovery by using the VirtualBox or UTM interface to power cycle the VM, boot normally (which may still go to rescue if the default is not restored), and follow the recovery steps above. Contact your instructor if you cannot restore normal boot.

Extra Credit — Cron Job Setup

+5 points

This section is optional. Completing it earns 5 extra credit points (CSLO 2, CSLO 3). It also gives you a useful head start on the Week 7 Server Configuration assignment, which requires a working cron job.

Cron is the traditional Unix job scheduler. It runs commands on a repeating schedule, independently of systemd. You will write a crontab entry, verify it runs, and explain the schedule expression.

Step 1 — Read the documentation

Before writing a crontab entry, review these free resources (they open in a new tab):

  • Ubuntu crontab(5) manpage — authoritative format reference
  • Crontab Guru — interactive schedule expression builder. Use this to test your expression before adding it to your crontab.
  • Use Cron to Automate Tasks on Ubuntu: Complete Guide

Step 2 — Design your cron job

Write a cron job that does something simple but verifiable. A good starting point is a job that appends the current date and time to a log file on a short, repeating schedule:

# Five time fields: minute hour day-of-month month day-of-week # This example runs every 5 minutes and appends to a log file */5 * * * * echo "$(date) -- cron ran" >> /home/yourusername/cron-test.log

Replace yourusername with your actual username. Use Crontab Guru to verify your schedule expression before adding it.

Step 3 — Add the job and verify it was saved

Open your crontab for editing (this opens your default text editor):

crontab -e

Add your cron job line at the end of the file. Save and exit. In nano: press Ctrl+O, then Enter to save, then Ctrl+X to exit.

Confirm the job was saved:

crontab -l

Take a screenshot of the crontab -l output showing your entry.

Step 4 — Verify the job ran

Wait for your schedule to trigger (if you used */5, wait at least 5 minutes), then check your log file:

cat /home/yourusername/cron-test.log

You should see one or more timestamped lines. Take a screenshot showing at least one entry. If you want to confirm using the system log instead:

grep CRON /var/log/syslog | tail -10
