Week 3

Runlevels, systemd, and Startup Scripts

This week you go under the hood to see how Linux boots and manages running services. You will learn where the traditional concept of runlevels came from, then shift into systemd, the init system you will find on virtually every modern Linux distribution. By the end of the week you should feel comfortable using systemctl to manage services, reading and writing a simple unit file, and reaching for journalctl when something does not start the way it should.

Learning Goals

After completing this week's work you should be able to:

  • 1 Trace the Linux boot sequence from firmware (BIOS/UEFI) through the bootloader and kernel to a working login prompt.
  • 2 Identify the traditional SysV runlevels 0 through 6 and describe the purpose of each one.
  • 3 Explain what systemd is and why it replaced SysV init on most modern distributions.
  • 4 Map traditional runlevels to their equivalent systemd targets, including multi-user.target and graphical.target.
  • 5 Use systemctl to start, stop, restart, reload, enable, disable, and check the status of a service.
  • 6 Describe the three sections of a .service unit file and write a simple one from scratch.
  • 7 Use journalctl with common flags to view and filter logs for a specific service, boot session, or time window.

Key Concepts This Week

SysV Init and Runlevels

The legacy init system defined system states with numbers 0 through 6. Knowing them is still useful because older documentation, scripts, and some tools still reference these numbers.

systemd and Units

systemd manages services, mounts, sockets, timers, and more through unit files. Each type has its own extension: .service, .target, .timer, and so on.

Targets vs. Runlevels

systemd replaces numeric runlevels with named targets. The two you will see most often are multi-user.target (roughly runlevel 3) and graphical.target (roughly runlevel 5).

systemctl

Your main tool for talking to systemd. Commands like systemctl status sshd, systemctl enable nginx, and systemctl set-default multi-user.target come up constantly in real admin work.

Unit File Structure

A .service file has three sections: [Unit] for description and ordering, [Service] for what to run, and [Install] for when to activate it. Getting these right avoids a lot of subtle startup problems.

journalctl and Logs

When a service fails silently, journalctl is usually the first place to look. Flags like -u sshd, -b (current boot), and --since "1 hour ago" help you narrow things down fast.

Looking Ahead

Coming up in Week 4: Process Management and Job Scheduling. Once you understand how services come to life at boot, the next question is what happens to those processes while the system is running. Week 4 covers the tools you use to inspect and control live processes: ps, top, htop, kill, and nice. You will also schedule recurring tasks with cron and at. The systemd knowledge from this week carries forward directly, because many modern systems now manage scheduled work through systemd timer units rather than traditional cron jobs.
