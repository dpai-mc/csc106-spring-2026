Week 3 Introduction: Runlevels and systemd Targets

In Week 2 you traced the Linux boot sequence from firmware and GRUB to the moment the kernel hands control to the first user-space process. This week picks up right there: that process is systemd, and it runs as PID 1. Week 3 is about understanding what systemd does when the system boots, how it replaced the older SysV init system, and how the traditional concept of runlevels maps onto the modern named-target model.

This is the first of two instructor videos for the week. It focuses on concepts and terminology. Video 2 is a hands-on terminal demo of the systemctl commands you will use in the systemd Exploration assignment.

Commands Demonstrated in This Video

These commands appear in the video in the order listed. Use this as a reference when following along or when completing the systemd Exploration assignment.

  1. ps -p 1 Confirm that systemd is running as PID 1 — the first user-space process started by the kernel after boot.
  2. systemctl get-default Display the current default boot target. On Ubuntu Server you will typically see multi-user.target.
  3. systemctl list-units --type=target List all currently loaded and active targets. Shows the checkpoints the system passed through during this boot.
  4. ls -la /lib/systemd/system/runlevel*.target View the compatibility symlinks that map old SysV runlevel numbers to their equivalent systemd target names.
  5. sudo systemctl set-default multi-user.target Change the default boot target. systemd implements this by updating a symlink at /etc/systemd/system/default.target. The change takes effect on the next boot.
  6. systemctl isolate <target> Switch the running system to a different target immediately, without rebooting. Mentioned in the video but not demonstrated live — see the caution note below before trying this.
Caution — systemctl isolate: This command stops all units not part of the target you switch to. Running isolate rescue.target will drop your system to a minimal single-user shell. Do not run this on the VM you use for the course unless you have a snapshot to restore from. If you want to experiment, create a separate throwaway VM first.
