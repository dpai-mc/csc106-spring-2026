CSC 106 — Week 1

Lab 1: Installing Ubuntu Server

Points: 25 Type: Hands-on Assignment CSLO 1

Scenario and Context

You have just been brought on as a junior systems administrator at a small nonprofit that runs its own website and a few internal tools. The organization's previous server was a shared hosting plan that kept going down and offered no control over the software stack. Leadership has approved a move to a self-hosted Linux server.

Your first task: stand up a fresh Ubuntu Server, confirm it is running correctly, and verify that you can install software on it. You will use this same server throughout the rest of the course — adding services, configuring the network, managing users, and eventually handing off a complete, documented server environment.

Why Linux? Linux is free, stable, widely used in server environments, and gives administrators fine-grained control over the software stack. The server edition (no graphical desktop) keeps the system lean, reduces the attack surface, and matches how real production servers are configured. When you connect to a remote server in a professional setting, this text-only environment is what you will see.

Phase 1 of 3

Prepare Your Virtual Machine

Before installing Ubuntu Server, you need a hypervisor — software that creates and runs virtual machines (VMs) on your computer. A VM lets you run a full Linux server inside a window on your existing system without needing dedicated hardware. This course supports two hypervisors:

  • VirtualBox — for Windows PCs (and Intel-based Macs)
  • UTM — for Macs with Apple Silicon (M1, M2, M3, M4, or later)
Before you continue: Follow the hypervisor setup guide for your computer (available in this week's Canvas module). Create a new VM and download the correct Ubuntu Server 24.04 LTS ISO for your platform. Attach the ISO to the VM. Return here once your VM is created and ready to boot from the ISO.
▶ Why the ISO architecture matters

Ubuntu Server 24.04 LTS comes in two builds for different CPU types:

  • amd64 (x86_64) — for VirtualBox on Windows or Intel Macs. Download the standard Ubuntu Server 24.04 LTS ISO.
  • arm64 (aarch64) — for UTM on Apple Silicon Macs. Download the Ubuntu Server 24.04 LTS for ARM ISO.

These two ISOs are not interchangeable. Attaching the wrong one will either fail to boot or run very slowly inside the VM. The setup guides for each platform tell you exactly which file to download.

▶ Recommended VM resource settings

These settings give you a functional server without requiring a high-end machine:

  • CPU: 2 virtual cores
  • RAM: 2 GB (2048 MB)
  • Virtual disk: 20 GB (dynamically allocated is fine)
  • Network adapter: NAT (default) — lets the VM reach the internet through your host machine

The hypervisor setup guides walk through these settings step by step.

Install Ubuntu Server 24.04 LTS

Take screenshots as you go. You need to capture at least three: the partitioning screen (before confirming), the user creation screen, and the first login prompt after the server finishes booting. You will upload these screenshots directly to the Canvas assignment alongside this completed lab.

Screenshot Checklist

Check each item as you capture it — these are the screenshots you will submit:

Did you use guided (automatic) or manual partitioning? Briefly describe what partitions the installer created — or what you created manually — and why you accepted or changed the defaults.

▶ What to look for: partitioning

Guided partitioning typically creates:

  • An EFI or boot partition — where the bootloader (GRUB) lives
  • A swap partition or swapfile — used as overflow memory when RAM is full
  • A root (/) partition — everything else; the main filesystem

For this course, the guided defaults are completely appropriate. Manual partitioning is used in production environments with specific storage layout requirements (for example, putting /var on its own partition to prevent log files from filling the root filesystem).

What hostname and username did you set? Why did you choose those names? If you used default suggestions, note what those were and whether you changed them.

Did you enable the OpenSSH server option during installation? Why does having SSH matter for a server with no graphical desktop?

▶ Why OpenSSH matters for a server

SSH (Secure Shell) is the standard tool for connecting to a Linux server remotely. Because Ubuntu Server has no graphical interface, SSH is often the only practical way to manage the server once it is in place — you connect from your laptop over an encrypted connection and run commands as if you were sitting in front of the machine.

Enabling SSH during installation means the service starts automatically at boot. You will put this to use in Week 7 when you configure SSH key-based authentication.

Phase 3 of 3

First Login and System Verification

The installation is complete and the server has rebooted. Now you will log in, confirm the system is running correctly, and verify that the package manager can reach the internet. These are the first commands a sysadmin runs on any freshly installed server.

Text-only login is normal. Ubuntu Server boots to a plain text prompt — no desktop, no mouse pointer. Type your username and press Enter, then type your password (characters will not appear as you type) and press Enter. If the login succeeds, you will see a shell prompt.
  1. 1

    Log in and note the shell prompt

    After entering your credentials, you will see a prompt that looks like this:

    yourusername@hostname:~$

    This prompt shows your username, the server hostname, and your current directory (~ is shorthand for your home directory). You are now inside the server. Every command you type will run directly on this Linux system.

  2. 2

    Check the kernel and system information

    Run this command exactly as shown:

    $ uname -a

    This prints a single line containing information about the running kernel and system. Paste the output below and answer the follow-up questions.

    Paste the full output of uname -a. Then, on separate lines below the output, identify: (1) the kernel version number, (2) the hostname, and (3) the CPU architecture — you should see x86_64 on VirtualBox or aarch64 on UTM.

    ▶ How to read uname -a output

    Example output:

    Linux myserver 6.8.0-45-generic #45-Ubuntu SMP PREEMPT_DYNAMIC Fri Sep 13 14:11:45 UTC 2024 x86_64 x86_64 x86_64 GNU/Linux

    • Linux — the kernel name
    • myserver — the hostname you set during installation
    • 6.8.0-45-generic — the kernel version number
    • x86_64 — the CPU architecture (you will see aarch64 on Apple Silicon)
    • GNU/Linux — the operating system type
  3. 3

    Confirm network connectivity

    Send four test packets to Google's public DNS server to verify the VM can reach the internet:

    $ ping -c 4 8.8.8.8

    Paste the complete output. Are packets being received (0% packet loss)? What does a successful ping confirm about the VM's network configuration?

    ▶ If ping fails or reports packet loss
    • Make sure the VM's network adapter is set to NAT in your hypervisor settings.
    • Check that your host machine is connected to the internet.
    • Try pinging your gateway instead: run ip route to find the gateway address, then ping -c 4 <gateway-address>. If that works but 8.8.8.8 does not, the issue may be DNS or a firewall on the host.
  4. 4

    Refresh the package index

    Run the command below. The sudo prefix runs the command with administrator privileges, which is required to update system-level package data. The first time you use sudo in a session you will be asked for your password.

    $ sudo apt update

    This does not install or upgrade anything — it contacts Ubuntu's package repositories and downloads a fresh list of what is available. You must do this before installing software on a fresh system.

    Paste the complete output of sudo apt update. How many package sources did apt check? Did it report any errors or warnings? In your own words, what did this command actually do?

    ▶ Reading apt update output

    Each line tells you what apt did with a specific repository entry from /etc/apt/sources.list:

    • Hit — apt checked the source; nothing has changed since the last update
    • Get — apt downloaded new package index data from this source
    • Ign — the entry was ignored (often a duplicate or unreachable source)

    The final summary line shows how many packages could be upgraded. Running apt update only refreshes apt's local index — it does not install or change anything. Think of it as refreshing the catalog before you shop.

  5. 5

    Reflect on the experience

    What surprised you about the installation process or the first login? Was there a step that was confusing or that you had to repeat? Is the text-only environment different from what you expected? Write at least two or three sentences.
